# Automy 奥托迈 · 文档与产品原型静态网站 Spec

> Change-ID: `automy-docs-prototype-site`
> 版本：v1.0
> 日期：2026-07-15
> 类型：新建静态网站（含细化文档 + 高保真原型）
> 关联前置：`enterprise-agent-platform`（平台整体需求规划，本变更聚焦"文档静态站点 + 产品原型"交付）

---

## 一、Why（项目背景与目标）

前置 spec `enterprise-agent-platform` 已完成 Automy 奥托迈（原名"有机系统"）平台整体需求规划。本变更不实现智能体平台本身，而是把**细化后的需求文档 + 技术架构设计文档 + 产品原型界面**整合为一个可部署到 Cloudflare Pages 的静态网站系统，作为：

1. 对内对齐：投资公司管理层、业务方、IT 部门统一认知的入口；
2. 立项展示：可在线浏览的高保真"可点击原型"，支撑立项评审与预算沟通；
3. 实施蓝本：后续真正开发 Automy 平台时，作为需求与设计的 single source of truth。

**关键调整（相对前置 spec）：**
- 系统名称：`有机系统 AgenticOS` → **`Automy 奥托迈`**
- 智能体编排框架：`LangGraph` → **`Eino ADK`（字节跳动开源的 Go 智能体开发框架）**
- 交付物范围：仅交付**静态文档站点 + 产品原型界面**，不交付 Automy 平台后端/前端实现代码。

---

## 二、What Changes（变更概览）

### 新建能力（本次交付）

- **静态网站系统（Automy Docs & Prototype Site）**
  - 纯前端静态站点，部署至 Cloudflare Pages；
  - 技术栈：HTML + CSS + 原生 JavaScript（用户偏好技术栈），可选轻量构建工具；
  - 多端响应式（桌面优先，平板适配）。

- **文档区（Docs）**
  - 需求文档（细化版，按模块组织）
  - 技术架构设计文档（细化版，含 Eino ADK 架构）
  - 路线图与里程碑
  - 术语表

- **产品原型区（Prototype）**
  - 高保真、可点击跳转的 HTML 原型；
  - 覆盖核心用户旅程的所有关键页面；
  - 模拟交互（侧边栏切换、Tab 切换、流式输出样式、市场卡片筛选、连接勾选等）；
  - 标注模式（可切换显示页面说明与需求条目编号）。

### 不在本次范围（明确排除）
- Automy 平台真实的后端服务、Agent 引擎、MCP 网关、模型网关、向量库等实现代码；
- Electron 桌面客户端实现（仅做 Web 原型，桌面端在原型中以"示意图"形式展示）；
- 真实 SSO、真实模型调用、真实数据持久化（原型用 Mock 数据）。

### 需求 1：站点整体结构与导航

#### REQ-1.1 站点信息架构
系统 SHALL 提供如下顶层导航：
- 首页（Overview）：项目简介、价值主张、核心能力一张图、进入文档/原型入口

- 技术架构（Architecture）
- 产品原型（Prototype）
- 路线图（Roadmap）
- 术语表（Glossary）

**Scenario: 首次访问**
- WHEN 访客打开站点首页
- THEN 看到 Automy 品牌标识、一句话定位、核心能力卡片（长期陪伴 / 技能市场 / 专家市场 / MCP 集成 / 安全合规）、CTA 进入文档或原型

#### REQ-1.2 全局视觉规范
系统 SHALL 统一品牌视觉：
- 品牌名：Automy 奥托迈；
- Logo 占位（首期文字 Logo：Automy）；
- 主色：可配置，建议深蓝/青色系（体现专业与可信）；
- 字体：中文系统字体栈 + 英文无衬线；

#### REQ-1.3 全站搜索
系统 SHALL 提供全站搜索（需求条目、架构章节、原型页面、术语），支持键盘快捷键（`/` 或 `Ctrl+K`）唤起。

### 需求 6：IM 通道集成与主动提醒（本次新增）

> 目标：让 Automy 从"被动应答"升级为"主动触达"，通过企业微信/飞书/钉钉等 IM 渠道主动提醒员工，并支持在 IM 内直接与 Agent 交互。

#### REQ-6.1 IM 通道接入
系统 SHALL 支持接入企业级 IM 通道作为通知与交互载体，首期支持：
- **企业微信**（WeCom Work）：应用消息、群机器人、客户消息回调
- **飞书**（Lark）：应用消息、群组消息、事件订阅
- **钉钉**（DingTalk）：工作通知、群机器人

每个通道 SHALL 支持配置：AppKey/AppSecret、回调地址、可见范围（部门/用户）、消息类型上限、频率限制。

**Scenario: 飞书通道接入**
- WHEN 管理员在 IM 通道配置页填写飞书应用凭证并完成 OAuth 授权
- THEN 平台建立与飞书的事件订阅通道，可在"已连接通道"中查看，并支持试发测试消息

#### REQ-6.2 统一通知服务
系统 SHALL 提供统一通知服务（Notification Service），屏蔽底层 IM 差异：
- 上游：技能/专家/编排引擎/审批流 通过统一 API 发起通知请求；
- 下游：按用户偏好与通道策略路由到 Web 站内信 / 企业微信 / 飞书 / 钉钉 / 邮件；
- 通知去重、合并、降级（如飞书失败降级到站内信）。

**Scenario: 任务完成多通道触达**
- WHEN 长时尽调任务在后台完成
- THEN 通知服务按用户偏好：飞书工作通知 + 站内信同时推送，并在飞书消息卡片中附带"查看报告"按钮，点击跳回 Automy 工作台对应会话

#### REQ-6.3 主动提醒引擎
系统 SHALL 提供主动提醒引擎，支持基于规则与基于事件的提醒触发：
- **场景触发**：任务完成、文档更新、会议即将开始、审批待办、记忆显著更新
- **时间触发**：日程提醒、日报周报提醒、长期未交互回访
- **规则编排**：可视化编排"触发条件 + 过滤条件 + 渠道选择 + 文案模板 + 去重策略"
- **节流策略**：每用户每日上限、单渠道上限、静默时段（如非工作时间降级为站内信）

**Scenario: 会议提醒**
- WHEN 用户日历显示 15 分钟后有"ABC 公司尽调会议"
- THEN 提醒引擎在会议前 15 分钟通过飞书发送提醒卡片，含会议要点摘要（基于会议相关记忆生成）+ 一键唤起工作台

#### REQ-6.4 通知模板与变量插值
系统 SHALL 提供通知模板管理：
- 模板字段：触发场景、渠道适配（各 IM 卡片结构不同）、标题、正文、操作按钮、变量占位符（如 {{user_name}} {{task_name}} {{report_url}}）
- 多语言支持；
- 在线试发与预览（各渠道卡片预览）。

#### REQ-6.5 双向交互（IM 内对话）
系统 SHALL 支持用户在 IM 端直接与 Agent 交互：
- 接收 IM 平台消息回调（用户@应用、点击卡片按钮）；
- 将消息转交 Agent 编排引擎处理；
- 响应结果回传 IM（文本/卡片/文件）；
- 会话上下文与 Web 端工作台打通（IM 发起的会话可在 Web 端续接）。

**Scenario: 飞书内直接提问**
- WHEN 员工在飞书群内 @Automy "查一下 CRM 里本部门上月商机"
- THEN 平台通过回调接收消息 → 调用 Agent → 经 MCP 连接 CRM 查询 → 将结果以卡片形式回传飞书群，并同步到 Web 端"我的会话"

#### REQ-6.6 通知权限与隐私
- 用户可配置个人通知偏好（渠道开关、静默时段、聚合策略）；
- 涉密内容仅站内信触达，不在 IM 显示正文（仅提示"有新的涉密通知，请登录查看"）；
- IM 通道调用全链路审计。

### 需求 7：会话数据库设计（本次新增）

> 目标：为多端会话、跨端续接、主动提醒提供数据底座。

#### REQ-7.1 会话数据模型
系统 SHALL 设计完整的会话数据模型，覆盖：
- **会话（Session）**：会话 ID、用户、标题、来源（Web/IM/桌面）、状态、关联项目、创建/最后活跃时间
- **消息（Message）**：消息 ID、会话、角色（user/assistant/tool/system）、内容、工具调用记录、token 统计、来源渠道
- **会话事件（SessionEvent）**：会话生命周期事件（创建/暂停/恢复/归档/移交）
- **会话参与者（SessionParticipant）**：多端/多 Agent 参与记录
- **主动推送记录（PushRecord）**：主动通知的发送记录、渠道、送达状态、用户交互（点击/忽略）

#### REQ-7.2 跨端续接
系统 SHALL 支持会话跨端续接：同一会话可在 Web/IM/桌面端切换并继续，所有端实时同步最新消息。

**Scenario: 飞书发起 → Web 续接**
- WHEN 员工在飞书发起会话提问，随后打开 Web 工作台
- THEN Web 端"我的会话"显示该会话（标注"飞书来源"），可继续追问，上下文无缝衔接

#### REQ-7.3 会话检索与归档
- 会话支持按用户/时间/项目/来源渠道检索；
- 长期未活跃会话自动归档（冷存储）；
- 归档会话可恢复，恢复后记忆上下文重新加载。

#### REQ-7.4 数据架构文档要求
架构文档区 SHALL 单独章节呈现会话数据库设计，包含：
- ER 图（Session/Message/SessionEvent/SessionParticipant/PushRecord）
- 存储分布（热数据 PG / 冷数据归档 / 消息检索向量库）
- 索引策略（用户+时间、会话+序号、来源渠道）
- 与记忆系统的关系（会话→记忆提取流水线）

### 需求 8：站点工程与部署

#### REQ-8.1 技术栈
- HTML5 + CSS3 + 原生 JavaScript（ES6+）；
- 构建：可选 Vite（仅做静态打包与开发服务器，不引入框架运行时）；
- 不引入 React/Vue 等框架运行时（保持轻量，对齐用户"静态网站开发"偏好）；
- 字体与图标使用本地资源或 CDN（可离线替换）。

#### REQ-8.2 目录结构
```
automy-site/
  index.html              # 首页
  /assets
    /css                  # 全局样式、组件样式
    /js                   # 全局脚本、搜索、主题切换、原型交互
    /img                  # 图片、Logo、占位
    /data                 # Mock JSON（技能、专家、MCP、记忆、审计等）
  /docs
    /requirements         # 需求文档各模块 HTML
    /architecture         # 架构文档各章节 HTML
  /prototype
    /auth                 # 登录引导
    /workspace            # 对话工作台及侧边栏
    /skill-market         # 技能市场与编辑器
    /expert-market        # 专家市场
    /mcp                  # MCP 目录与连接
    /knowledge            # 知识库
    /profile              # 用户画像
    /admin                # 管理后台
  /shared
    /components           # 复用组件（按钮、卡片、表格、侧边栏、顶栏）
  404.html
  wrangler.toml           # Cloudflare Pages 配置（可选）
  README.md              # 运行与部署说明（用户需求，可保留）
```

#### REQ-8.3 部署
- 一键部署至 Cloudflare Pages；
- 提供本地预览命令（`npx serve` 或 `vite dev`）；
- 支持 `wrangler pages deploy`。

#### REQ-8.4 性能
- 首屏 LCP < 2s；
- 单页 JS < 50KB（压缩后，不含字体）；
- 文档与原型页面懒加载（按需加载 HTML 片段或整页跳转）。

#### REQ-8.5 可访问性
- 语义化 HTML；
- 键盘可导航；
- 颜色对比度 AA。

## 五、技术架构建议（针对静态网站本身）

> 说明：本节为"静态网站"的技术架构，非 Automy 平台本身的架构（后者见架构文档区）。

### 5.1 站点架构

```
┌──────────────────────────────────────────────────┐
│  Cloudflare Pages (CDN + HTTPS)                  │
├──────────────────────────────────────────────────┤
│  静态资源 (HTML / CSS / JS / Mock JSON / 图片)   │
├──────────────────────────────────────────────────┤
│  前端运行时                                       │
│  ┌────────────┐ ┌──────────┐ ┌────────────────┐ │
│  │ 路由(Hash) │ │ 主题切换 │ │ 全站搜索       │ │
│  └────────────┘ └──────────┘ └────────────────┘ │
│  ┌────────────┐ ┌──────────┐ ┌────────────────┐ │
│  │ Mermaid   │ │ 代码高亮  │ │ 原型交互脚本   │ │
│  └────────────┘ └──────────┘ └────────────────┘ │
└──────────────────────────────────────────────────┘
```

### 5.2 关键设计要点

1. **零后端**：纯静态，所有"数据"来自 `/assets/data/*.json` Mock 文件。
2. **路由**：使用 Hash 路由（`#/docs/...`、`#/prototype/...`），便于 Cloudflare Pages 直接托管，无需 rewrite 规则。
3. **组件复用**：顶栏、侧边栏、卡片、表格、需求条目卡等抽取为 HTML 片段 + JS 渲染函数，避免重复。
4. **文档渲染**：Markdown 文档可预先用构建脚本转为 HTML 片段，或直接手写 HTML（推荐手写以获得最佳视觉控制）。
5. **图表**：Mermaid.js 通过 CDN 引入，按需渲染。
6. **主题**：CSS 变量驱动深浅色切换，localStorage 持久化。
7. **搜索**：构建期生成索引 JSON（标题 + 正文摘要 + 锚点），运行时前端检索。
8. **可演化**：文档与原型分离目录，便于后续独立更新。

### 5.3 与 Automy 平台架构的关系

本静态网站**不实现** Automy 平台架构，但**呈现** Automy 平台架构。架构文档区需准确反映未来 Automy 平台将采用的架构（含 Eino ADK），以便评审方理解。

## 八、风险与对策

| 风险 | 对策 |
|------|------|
| 文档与原型信息不一致 | 需求条目编号双向链接，原型标注模式可校验 |
| Eino ADK 资料有限、架构描述偏差 | 架构文档标注"基于公开资料推断"，留待实施期校准 |
| 原型页面过多、维护成本高 | 组件化 + Mock 数据驱动，减少重复 HTML |
| 静态站点搜索体验弱 | 构建期预生成索引，前端轻量检索即可满足 |
