    • 知识库与 RAG

    检索增强生成(RAG)是 Automy 知识库的核心技术,让 Agent 能精准引用企业内部文档,提升回答准确性与可溯源性。 本章节面向投资公司高密度文档场景(财报、研报、尽调材料、法规、合同),介绍文档处理流水线、混合检索架构、 Embedding 选型、引用溯源设计、增量更新与权限隔离。

    📚
    设计目标
    回答可溯源到文档级 · 多模态文档(表格/图)结构化保留 · 混合检索召回率 >90% · 数据权限按部门/项目隔离

    一、RAG 整体架构

    Automy RAG 系统采用「离线写入 + 在线检索」双链路设计。离线链路负责文档接入、解析、分块、向量化、入库; 在线链路负责查询改写、混合检索、重排序、上下文组装、引用生成。两条链路通过共享知识索引解耦,支持文档增量更新不阻塞在线服务。

    二、文档处理流水线

    文档处理流水线是 RAG 质量的基石。投资公司文档形式复杂(扫描版财报、含图表的研报、嵌套表格的合同), 流水线需保留语义结构,避免「分块切断表格」导致信息丢失。

    2.1 文档解析(关键环节)

    文档类型解析方案处理要点
    PDF(数字版)PyMuPDF + pdfplumber保留书签、表格抽取向量化
    PDF(扫描版)PaddleOCR + 版面分析OCR 文字识别 + 表格结构还原
    Word/Excelpython-docx / openpyxl保留样式、公式、合并单元格
    PPTpython-pptx按 Slide 分块、保留备注
    图片(JPG/PNG)多模态模型(VLM)图表理解生成描述,图像入向量库

    2.2 智能分块策略

    分块大小直接影响召回精度。Automy 采用「语义优先 + 兜底窗口」策略:

    • 语义边界分块:按段落、标题层级切分,块大小 300-800 tokens
    • 表格整体保留:表格不切分,作为独立 chunk,附加列名摘要
    • 重叠窗口:相邻块保留 50-100 tokens 重叠,避免上下文断裂
    • 父子分块:小块用于检索,父块(含上下文)用于喂给 LLM
    • 层级标题路径:每块携带 H1/H2/H3 路径,辅助理解

    三、混合检索架构

    单一向量检索在精确匹配(股票代码、法规编号、合同条款号)场景表现差, 单一关键词检索无法理解语义。Automy 采用三路混合 + 重排序架构,综合语义、关键词、元数据三维信息。

    3.1 检索策略对比

    策略原理优势短板适用场景
    向量检索(Dense)Embedding 余弦相似度理解语义、抗拼写误差精确匹配弱语义相似查询
    BM25(Sparse)TF-IDF 改进精确匹配、术语命中无法理解同义代码、股票代码、法规号
    元数据过滤结构化字段筛选权限隔离、范围限定不能解决语义权限控制、时间范围
    RRF 融合排名倒数加权无需调参、综合两路需精排补充默认召回策略
    Cross-Encoder RerankQuery-Doc 联合编码精度最高计算开销大Top-K 精排

    3.2 查询改写(提升召回)

    • Query Rewriting:LLM 改写用户口语化查询为规范化检索 query,扩展同义词
    • HyDE(Hypothetical Document Embedding):先让 LLM 生成假设性答案,用答案向量检索(适合长尾问题)
    • Multi-Query:LLM 生成 3-5 个不同视角的子查询,并行检索后融合
    • 查询路由:根据查询类型(事实型/分析型/列表型)选择不同检索参数

    四、Embedding 模型选型

    Embedding 模型决定向量检索上限。Automy 私有化部署,需在「中文效果 + 多语言 + 部署成本」之间平衡。

    模型维度中文多语言长度许可推荐场景

    BGE-large-zh-v1.51024⭐⭐⭐⭐⭐✗512MIT纯中文场景
    m3e-base768⭐⭐⭐⭐弱512Apache-2.0通用中文

    ✅
    选型决策:BGE-M3
    支持稠密向量 + 稀疏向量 + 多向量三合一,长文本 8192 tokens,中文 MTEB 排名前列,MIT 许可可商用,显存占用 ~2GB,适合单卡部署。

    五、引用溯源设计

    引用溯源是投资公司 RAG 的硬性要求:分析师撰写报告时,每个数据点必须可追溯到原始文档与页码,便于复核与合规审计。 Automy 设计了「块级 + 页级 + 文档级」三级引用标注。

    六、增量更新与索引维护

    文档会更新、修订、废止。RAG 系统必须支持增量更新,避免「全量重建」带来的服务中断与算力浪费。

    文档增量更新流程 Mermaid

    (SHA-256)"] HASH -->|哈希一致| SKIP["跳过,无需更新"] HASH -->|哈希变化| DIFF["Diff 计算

    记忆失效"] style SKIP fill:#E5E7EB,stroke:#6B7280 style UPD fill:#D1FAE5,stroke:#059669 style NOTIFY fill:#FEF3C7,stroke:#D97706

    七、权限隔离设计

    投资公司文档高度敏感(未公开的尽调材料、内部研究),RAG 检索必须强制按权限过滤,避免越权访问。 Automy 在检索前预过滤(而非检索后过滤),保证低权限用户无法接触到敏感向量。

    维度过滤机制实现
    部门用户部门 ↔ 文档部门Milvus metadata filter: dept_id IN (用户部门+祖先)
    项目项目成员 ↔ 文档项目project_id IN (用户参与项目列表)
    密级用户密级 ≥ 文档密级access_level <= user_clearance
    角色RBAC 资源权限PG ACL 表查询
    时间临时授权窗口auth_window 字段校验

    九、业务场景示例:投研尽调

    📋 场景:分析师撰写「ABC 新能源」尽调报告
    1. 分析师提问:「ABC 新能源近三年毛利率变化趋势及驱动因素?」
    2. 查询改写:扩展为「ABC 新能源 毛利率 2022 2023 2024 2025 同比 变动 原因」
    3. 三路检索:向量(语义相似) + BM25(精确年份/科目) + 元数据(部门=投研、密级≤L3)
    4. RRF 融合 Top-50 → Rerank 精排 Top-5(财报 + 研报 + 行业数据)
    5. LLM 生成带 [1][2][3] 引用的分析,每个数据点带文档+页码
    6. 分析师点击 [1] 验证 → 弹出 ABC 财报第 12 页原文高亮
    7. 导出报告时,引用自动转为脚注,符合合规审计要求
    ⚠
    已知的局限与对策
    表格复杂场景:跨页大表分块易丢失结构 → 父子分块 + 表格摘要补充
    多模态图表:折线图/柱状图数据无法精确召回 → VLM 生成图表描述入向量库
    长文档全局语义:单块检索缺乏全局视角 → 文档级摘要单独索引
