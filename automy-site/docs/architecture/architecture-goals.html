    本章节是技术架构的纲领性文档,从战略层面定义 Automy 奥托迈平台的考核维度、核心实体关系、双系统架构形态、框架选型依据与企业级分布式能力。 所有后续章节(架构总览、系统设计、AgentScope Java 2.0 运行时、记忆系统等)均围绕本目标展开,确保技术决策有据可依、演进方向一致。

    文档定位
    架构目标 = 考核四要素 + 实体关系 + 双系统架构 + 框架选型 + 企业级分布式 · 是所有技术架构章节的纲领

    一、考核四要素

    Automy 作为面向投资公司的企业级智能体平台,其架构设计必须接受四个维度的考核。这四个要素相互制约、彼此平衡,共同构成平台技术决策的评判标尺。

    ⚖️
    考核原则
    效果是根本 · 效率是体验 · 质量是底线 · 成本是约束 · 四者动态平衡,在不同阶段侧重不同
    ① 效果 Effectiveness 根本

    智能体能否真正完成用户委托的任务,产出结果是否准确、可用、可信赖。这是平台存在的根本价值。

    核心指标

    • 任务完成率(Task Success Rate) ≥ 85%
    • 工具调用准确率(Tool Call Accuracy) ≥ 95%
    • RAG 检索命中率(Hit Rate) ≥ 90%
    • 记忆召回相关率(Recall@10) ≥ 80%
    • 用户满意度评分(CSAT) ≥ 4.3 / 5

    关键手段

    • 分层记忆 + 长期记忆沉淀用户偏好与事实
    • RAG 知识库 + 混合检索(向量+关键词+元数据)
    • MCP 协议标准化接入企业真实系统
    • 多 Agent 协同 + 工作流编排处理复杂任务
    • Function Calling + 工具结果缓存
    ② 效率 Efficiency 体验

    在效果达标的前提下,响应速度与资源利用效率决定用户体验。投资业务节奏紧凑,慢一拍即失先机。

    核心指标

    • 首字响应时间(TTFT) < 800ms
    • 整句响应时间(P95) < 3s
    • 工具调用平均延迟 < 500ms
    • GPU 利用率 ≥ 70%
    • 并发会话吞吐 ≥ 100 QPS / 节点

    关键手段

    • Spring Boot 微服务 + gRPC 内部通信
    • LLM 流式输出(SSE)+ KV Cache 复用
    • 无依赖步骤并行执行
    • Redis 缓存热点上下文与工具结果
    • vLLM 推理池 + 连续批处理
    ③ 质量 Quality 底线

    系统稳定性、可观测性、安全合规与可维护性,是企业级平台不可触碰的红线,金融场景尤甚。

    核心指标

    • 系统可用性(SLA) ≥ 99.9%
    • 错误率 < 0.1%
    • 全链路可观测覆盖率 100%
    • 审计日志完整率 100%
    • 数据安全合规(等保三级 + 金融监管)

    关键手段

    • 分布式架构 + 高可用容灾
    • Prometheus + Loki + OpenTelemetry 三位一体可观测
    • Langfuse LLM 可观测与质量分析
    • RBAC + ABAC 双重权限 + 全链路审计
    • 关键操作签名链防篡改
    ④ 成本 Cost 约束

    在效果、效率、质量均达标的前提下,单位算力与运维成本是否可控。投资公司需量化 AI ROI。

    核心指标

    • 单次会话平均 Token 成本
    • GPU 单卡日均处理会话数
    • 人均月度算力消耗
    • 运维人力成本 / 100 用户
    • 总拥有成本(TCO)3 年回收期

    关键手段

    • 本地推理(vLLM)避免 API 调用费用
    • 模型分级路由(简单→小模型,复杂→大模型)
    • KV Cache 复用 + 推理批处理
    • 记忆冷热分层 + 向量库压缩
    • 技能 / 专家市场复用避免重复开发

    四要素的动态平衡

    在不同业务阶段,四要素的侧重不同。架构设计必须为这种动态平衡预留弹性:

    阶段侧重取舍说明
    POC / 试点效果 > 效率先证明智能体能完成任务,延迟可以稍高
    规模化推广效率 > 成本用户量上来后,响应速度与并发能力是关键
    生产稳定期质量 > 一切金融业务场景下,稳定与合规是底线
    降本增效期成本 > 效率在效果质量不变前提下,压缩算力与运维成本

    运行时加载流程

    Harness 框架是通用的执行内核,本身不携带任何业务逻辑。每个用户选中某个智能体后,实际运行时由 Harness 加载对应智能体及其相关参数、技能、MCP 工具等,形成运行态实例。

    核心设计理念

    1. Harness 框架通用化

    Harness(Agent Runtime Kernel)是平台唯一的执行内核,与具体智能体解耦。它只负责加载、调度、记忆、工具路由等通用能力,不携带任何业务 Persona。所有个性化通过运行时加载的智能体参数注入。

    2. 一人一默认智能体

    每个用户有且仅有一个"默认智能体",作为 IM 通道主动联系时的对话对象。用户可订阅多个智能体,但默认智能体唯一,便于 IM 通道的统一入口设计。默认智能体可随时切换。

    3. 分布式定时任务主动触达

    智能体可注册分布式定时任务(Cron + Redis RedLock + NATS 队列),到点主动触发,通过 IM 通道通知用户。如:每日盘前 8:00 推送持仓分析、季报披露日提醒等。

    4. IM 通道双向交互

    IM 通道不仅用于接收定时任务的主动推送,也支持用户主动联系默认智能体。用户在飞书/企微中 @ 智能体即可发起对话,消息回流到 Harness 执行,响应通过同一 IM 通道返回。

    5. 技能与 MCP 解耦

    技能是平台内的高层抽象(用户可自定义),MCP 是对接企业系统的低层标准协议。技能可以调用 MCP 工具,但 MCP 工具不等于技能。一个技能可组合多个 MCP,一个 MCP 也可被多技能复用。

    6. 订阅即装配

    用户订阅智能体的过程即装配:智能体定义 + 技能列表 + MCP 绑定 + 知识库挂载 + 定时任务注册,形成完整运行态配置。订阅后用户可在自己空间内进一步安装额外技能与 MCP。

    2. 工作系统 AI Agent Hub(智能体中台)

    定位:强调"连接、分发、执行"的运行态容器。内部基于一套公共的 Harness 框架(Agent Runtime Kernel)的 Agent 来工作。用户选中某个智能体后,实际运行时就加载对应的智能体及相关参数、技能和 MCP 工具等。

    这两套系统在工程上有严格的边界,通过共享存储(数据库)解耦,任何产物必须先入库才能被 runtime 加载,避免直接内存传递带来的不可控。

    3. 选型结论与理由

    ✅
    最终选型:AgentScope Java 2.0 作为 Harness 框架实现
    AgentScope Java 2.0 在生产就绪度、性能、生态成熟度上综合得分最高,作为 Automy 的 Harness 框架(Agent Runtime Kernel)实现
    选型理由说明
    生产就绪阿里达摩院开源,Spring 生态稳定,适合企业级场景
    性能优势Java 原生(JDK 17+) + Project Reactor 响应式非阻塞高并发,符合"效率"考核要素
    声明式编排ReActAgent + Supervisor 多 Agent 编排直观,支持复杂工作流
    MCP 原生支持与 Automy 的 MCP 协议战略一致,无需额外封装
    可观测原生内置 OpenTelemetry,与 Prometheus / Loki / Jaeger 一体化

    4. LangChain 生态集成

    AgentScope Java 2.0 是 Java 原生框架,直接嵌入 Spring 生态,同时通过 MCP 协议桥接 Python 工具生态,复用 LangChain 丰富的工具与 Chain 实现,避免重复造轮子。

    1. 企业级数据库存储会话与记忆

    会话记录与用户记忆必须持久化到企业级数据库,避免单机故障导致数据丢失。这是企业级平台与单机 Demo 的本质区别。

    存储类型选型用途高可用方案
    关系数据库PostgreSQL 15+会话记录、用户、智能体定义、技能、审计主从复制 + 读写分离 + 自动故障转移
    向量数据库Milvus 2.x 集群长期记忆向量、RAG 知识向量分布式分片 + 副本
    缓存Redis 7 哨兵会话上下文、限流、分布式锁主从 + 哨兵自动切换
    对象存储MinIO 集群文档、附件、技能包、备份纠删码 + 多副本
    消息队列NATS JetStream异步事件、任务队列、定时任务集群 Raft 共识

    2. 便于排查用户问题与诊断故障

    分布式存储的另一关键价值:全链路数据沉淀,便于事后排查用户问题与诊断系统故障。

    用户问题排查

    • 会话回放:通过 session_id 查询 PostgreSQL 完整对话历史
    • 上下文还原:Redis 保留最近会话上下文,可重现场景
    • Trace 追踪:Jaeger 按 trace_id 串联整条调用链
    • LLM 调用回放:Langfuse 记录每次模型调用的输入输出
    • 工具调用审计:审计库记录所有 MCP 工具调用参数与结果

    系统故障诊断

    • 指标告警:Prometheus 实时监控 QPS / 延迟 / 错误率
    • 日志聚合:Loki 按 label 检索全集群日志
    • 链路追踪:OpenTelemetry 跨服务 Trace 串联
    • 事件溯源:NATS JetStream 持久化事件,可重放
    • 审计签名链:关键操作签名链防篡改,支持司法取证

    4. 分布式定时任务架构

    智能体的主动触达能力依赖分布式定时任务。Automy 采用 Redis RedLock + NATS 队列的组合方案,确保定时任务在集群环境下不重复执行、不遗漏执行。

    sequenceDiagram participant CRON as 定时触发器 participant REDIS as Redis (RedLock) participant NATS as NATS 队列 participant AG as Agent Runtime participant IM as IM 通道 participant U as 用户 CRON->>REDIS: 尝试获取分布式锁 REDIS-->>CRON: 锁获取成功(唯一节点) CRON->>NATS: 投递任务消息 NATS->>AG: 消费任务 AG->>AG: 执行智能体逻辑 AG->>IM: 推送通知 IM->>U: 主动触达用户 AG->>NATS: ACK 任务完成 CRON->>REDIS: 释放锁
    设计要点实现方案价值
    分布式锁Redis RedLock 算法避免集群中多节点同时触发同一任务
    任务持久化NATS JetStream任务不丢失,支持重试与死信队列
    幂等性任务 ID + 去重表重复消费不产生副作用
    可观测任务执行 Trace + 日志每个定时任务可追踪、可审计
    失败重试NATS 重试策略 + 告警网络抖动等可自动恢复
