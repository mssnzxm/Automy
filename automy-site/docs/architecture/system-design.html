    • 系统设计与分层

    本章节深入 Automy 五层架构的内部,详细说明模块划分、服务间通信、API 设计规范与端到端数据流。 系统设计的目标是让每个微服务职责单一、边界清晰、协议标准,从而支撑投资公司高强度的业务负载与严格的合规要求。

    🧩
    设计准则
    单一职责 · 接口契约先行 · 同步走 gRPC、异步走 NATS · 流式响应用 SSE · 全链路可观测

    二、各层详细职责

    L5 用户体验层

    面向最终用户与管理员的 Web 界面。技术选型 React + Vite + Tailwind + shadcn/ui,通过 REST API 与应用服务层通信,支持 SSE 流式响应以呈现 LLM 逐字输出。

    模块主要功能关键交互
    工作台 Workspace会话管理、项目空间、多 Agent 切换REST + SSE 流式
    技能市场 Skill Market技能浏览、安装、订阅、评分REST
    专家市场 Expert Market专家订阅、配置、试用REST
    用户中心 Profile个人偏好、记忆管理、API TokenREST
    管理后台 Admin用户/角色、审计、监控、系统配置REST

    L4 应用服务层

    业务逻辑编排层,采用 Java(Spring Boot)微服务架构,每个服务独立部署,通过 gRPC/REST 互相调用,通过 NATS 异步解耦。

    • Session Service:会话创建、上下文维护、消息路由、SSE 长连接管理
    • Skill Engine:技能调度、参数校验(JSON Schema)、版本管理、执行编排
    • Expert Engine:专家配置加载、Persona 组装、能力组合、订阅计费
    • Doc Service:文档上传、解析任务排队、分块管理、增量索引
    • Audit Service:操作记录、签名链、合规报告、独立存储
    • Notify Service:邮件、IM(Webhook)、站内消息、待办推送

    L3 智能体核心层

    Automy 的核心层,基于 AgentScope Java 2.0 实现。详见 智能体运行时章节。

    • AgentScope Runtime:ReActAgent 推理内核 + HarnessAgent 工程化底座,Supervisor 多 Agent 编排,响应式流式调度
    • Workspace:基于 AbstractFilesystem 的工作空间(本地/Docker/对象存储),按 sessionId/userId 隔离
    • Memory:MEMORY.md 自动管理 + 自动合并/压缩,分层记忆(短期/工作/长期)
    • RAG Service:knowledge/ + RAG 索引,混合检索(向量+关键词+元数据)、引用溯源
    • MCP Hub:@Tool 注解 + Toolkit,对接 MCP Tool Hub,Nacos 服务发现

    L2 数据与模型层

    组件用途选型
    向量数据库RAG 检索、记忆向量Milvus 2.x(HNSW/IVF)
    关系数据库用户/会话/配置/审计PostgreSQL 15+(JSONB/分区表)
    对象存储文档/附件/备份MinIO(S3 兼容)
    缓存会话上下文/限流/锁Redis 7
    模型网关多模型路由/降级/配额自研 + LiteLLM
    本地推理私有化 LLMvLLM(Qwen / DeepSeek / GLM)

    L1 基础设施层

    • Keycloak:SSO / OIDC / SAML 集成,统一身份认证
    • Kong:API 网关,统一入口、JWT 鉴权、限流、灰度
    • NATS:消息队列,JetStream 持久化,异步任务与事件总线
    • Prometheus + Grafana:指标采集与可视化告警
    • Loki:日志聚合,与 Grafana 一体化查询
    • OpenTelemetry + Jaeger:分布式链路追踪

    四、API 设计规范

    Automy API 遵循 RESTful 规范,所有 LLM 交互场景支持 SSE 流式响应。版本通过 URL 前缀 /api/v1 标识。

    RESTful 规范

    方法语义示例
    GET查询资源GET /api/v1/sessions
    POST创建资源POST /api/v1/sessions
    PATCH部分更新PATCH /api/v1/sessions/{id}
    DELETE删除资源DELETE /api/v1/sessions/{id}
    POST + SSE流式动作POST /api/v1/sessions/{id}/messages

    统一响应格式

    JSON - 统一响应结构

    SSE 流式响应

    LLM 输出与长任务进度通过 SSE(Server-Sent Events)推送到前端,事件类型按业务语义划分:

    event: token
data: {"content": "营收"}

event: token
data: {"content": "12 亿"}

event: tool_call
data: {"tool": "crm.opportunity.list", "args": {...}}

event: citation
data: {"document_id": "doc-001", "page": 12}

event: done
data: {"message_id": "msg-...", "tokens": 1024}

    gRPC 内部接口

    内部服务间使用 gRPC + Protobuf 通信,关键 proto 定义:

    service AgentRuntime {
  // 同步流式调用
  rpc Invoke(InvokeRequest) returns (stream InvokeChunk);
  // 异步任务(返回 task_id)
  rpc SubmitTask(TaskRequest) returns (TaskAck);
  rpc CancelTask(CancelRequest) returns (CancelAck);
}

message InvokeRequest {
  string session_id = 1;
  string expert_id = 2;
  string prompt = 3;
  map<string, string> params = 4;
  Context ctx = 5;
}

    数据流关键节点说明

    阶段关键动作数据存储
    1. 接入Kong JWT 校验、限流配额、路由到 Session ServiceRedis(限流计数)
    2. 上下文加载Session Service 加载会话历史、用户偏好PostgreSQL + Redis
    3. 记忆检索AgentScope Runtime 调用 Memory Service 检索 Top-K 相关记忆Milvus + PostgreSQL
    4. 知识检索若涉及企业文档,触发 RAG 检索混合召回Milvus + MinIO
    5. 工具调用Function Calling 调用 MCP 工具(如 CRM 查询)外部企业系统
    6. 模型推理Model Gateway 路由到 vLLM,流式返回 TokenGPU 推理池
    7. 流式输出Token 逐字回流到前端,前端实时渲染SSE 长连接
    8. 记忆写入异步提取本次会话的关键事实/偏好写入记忆Milvus + PostgreSQL
    9. 审计落库全程事件投递 NATS,Audit Service 消费落库独立审计库 + MinIO

    六、错误处理与降级

    🛡 分级错误处理

    • L1 参数错误:4xx 返回,前端提示
    • L2 业务错误:5xx + 错误码,记录日志
    • L3 依赖故障:熔断 + 降级 + 重试
    • L4 系统故障:告警 + 自动恢复

    🔄 降级策略

    • 模型不可用 → 切换备用模型(Qwen → DeepSeek → GLM)
    • RAG 失败 → 直接走 LLM,标注无引用
    • MCP 工具超时 → 返回提示而非阻塞
    • 记忆写入失败 → 重试 + 死信队列
