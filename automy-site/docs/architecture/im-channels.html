    • IM 通道集成架构

    IM 通道是企业微信、飞书、钉钉三大办公协同平台与 Automy 智能体之间的接入与交互载体。本章节定义通道适配器层、统一通知服务、主动提醒引擎、双向交互回调、降级与节流、安全审计、技术选型等核心设计, 让 Automy 从被动应答升级为主动触达与双向交互,真正融入投资公司日常投研、风控、合规工作流。

    📨
    设计原则
    通道适配器屏蔽 IM 差异 · 统一 API 上游 · 降级链保障可用性 · 节流避免打扰 · 全链路审计可追溯 · 涉密内容仅站内信

    一、概述

    投资公司员工日常高频使用企业微信、飞书、钉钉进行沟通协作,智能体若仅能在 Web 工作台使用,存在"打开成本高、被动等用户、消息孤岛"等问题。Automy 通过 IM 通道集成实现从被动应答到主动触达的转变:上游业务事件(任务完成、文档更新、会议开始、审批待办、记忆更新)经主动提醒引擎规则匹配与节流后,通过统一通知服务路由到通道适配器,最终投递到员工所在 IM 平台;IM 内 @应用与卡片按钮回调可路由到 Agent 编排引擎处理,会话上下文与 Web 工作台打通实现跨端续接。

    二、通道接入架构

    通道接入采用三层架构:统一通知服务对接上游业务系统,通道适配器层屏蔽各 IM 平台差异,IM 平台提供底层通信能力。每个 Adapter 实现发消息、接收回调、卡片渲染适配三类能力。

    三、统一通知服务设计

    统一通知服务是上游业务与下游通道之间的中间层,屏蔽 IM 平台差异,提供一致的上游 API 与路由策略。

    上游统一 API 
    // 统一通知 API
type NotifyRequest = {
  userId: string            // 目标用户
  scene: string             // 触发场景(report.completed/meeting.start)
  payload: object          // 模板变量数据
  channels?: Channel[]      // 可选,指定渠道
  priority?: Priority       // P0 立即 / P1 延迟 / P2 仅站内信
}

// 路由策略
1. 读取用户偏好(渠道开关 / 静默时段 / 聚合)
2. 渠道选择(指定 > 偏好 > 默认)
3. 降级链(飞书 → 企微 → 站内信)
4. 去重(同场景同内容 5 分钟一次)
5. 合并(同用户 1 分钟窗口聚合)
6. 节流(每用户每日上限 20 条 / 静默时段)

    📨 去重

    • 同场景同内容 5 分钟内仅投递一次
    • 按 user_id + scene + content_hash 去重
    • 重复请求合并为已读状态更新

    📚 合并

    • 同用户多场景通知 1 分钟窗口内合并
    • 合并为聚合卡片展示多条
    • 保留每条通知的可点击按钮

    🚦 节流

    • 每用户每日上限默认 20 条
    • 静默时段(21:00-08:00)仅紧急
    • 同场景 1 小时内合并

    🔀 降级

    • 渠道失败率 >10% 自动降级
    • 降级链:飞书 → 企微 → 站内信
    • 涉密内容强制降级到站内信

    四、主动提醒引擎时序

    主动提醒引擎是触发源到通知服务的中间编排层,负责规则匹配、模板渲染、节流控制。下图展示从触发源到用户点击回调的完整时序。

    五、双向交互回调流程

    双向交互支持 IM 内 @应用与点击通知卡片按钮两种方式,回调经 Webhook 接收后路由到 Agent 编排引擎处理,关键点是会话上下文与 Web 端打通,来源渠道标注。

    双向交互回调时序图 Mermaid
    sequenceDiagram participant U as 用户 participant IM as IM 平台(飞书) participant WH as Webhook 接收器 participant ROUTE as 会话路由 participant AGENT as Agent 编排引擎(Eino ADK) participant MCP as MCP 工具(CRM/知识库) participant RENDER as 结果渲染器 participant ADAPTER as IM Adapter participant WEB as Web 工作台 U->>IM: @Automy 查询"ABC 新能源估值" IM->>WH: 事件回调(消息内容+用户身份) WH->>WH: 签名校验 + SSO 鉴权 WH->>ROUTE: 路由请求 ROUTE->>ROUTE: 查找/创建 Session(source=im_lark) ROUTE->>AGENT: 调度 Agent 处理 AGENT->>MCP: 调用 CRM MCP 工具 MCP-->>AGENT: 返回 ABC 估值数据 AGENT->>RENDER: 生成回答 RENDER->>ADAPTER: 渲染飞书卡片 ADAPTER->>IM: 发送卡片回答 IM-->>U: 展示回答卡片 ROUTE->>WEB: 同步会话与消息(标注来源飞书) Note over U,WEB: 用户可在 Web 工作台继续追问,上下文打通
    环节关键设计异常处理
    Webhook 接收校验签名(防伪造)+ SSO Token 鉴权签名失败丢弃并告警
    会话路由查找最近活跃 Session(同用户同主题),否则新建会话不存在则新建并标注 source_channel
    Agent 编排基于 Eino ADK Graph 编排,可调用 MCP 工具工具调用失败降级返回提示
    结果渲染渲染为 IM 卡片(飞书 interactive / 企微 markdown)渲染失败退化为文本
    跨端同步同步到 Web 工作台,标注来源渠道同步失败不影响 IM 端回答

    七、安全审计

    🔐 凭证加密

    • AppKey / AppSecret 通过 KMS 加密存储
    • 数据库字段级加密(AES-256)
    • 90 天自动轮换提醒
    • 凭证访问全审计

    ✍ 签名校验

    • Webhook 回调强制签名校验
    • 飞书:事件订阅 v2 签名
    • 企微:回调 Token + AES 解密
    • 钉钉:Stream 模式 + 签名

    📜 内容审计

    • IM 调用全链路审计日志
    • 记录 user / scene / channel / status
    • 内容摘要加密存储
    • 合规审计员可按维度查询导出

    🔒 涉密过滤

    • 涉密关键词词典(项目代号等)
    • 消息投递前 NLP 脱敏
    • L3+ 内容强制站内信
    • 审计日志记录降级原因
