    • 部署与运维

    Automy 支持完整的私有化部署,基于容器化与 Kubernetes,提供从开发到生产的一站式部署方案。 本章节介绍部署拓扑、K8s 资源规划、三种部署模式(单机/集群/离线)、可观测性栈、 CI/CD 流水线、灾备与回滚,所有设计满足投资公司「数据不出内网」的硬性要求。

    ☁
    部署目标
    私有化优先 · 零外网依赖 · 弹性扩缩容 · 滚动零停机 · RTO < 4h / RPO < 5min

    一、生产部署拓扑

    生产环境采用 Kubernetes 集群部署,所有组件容器化(基于 JDK 17+ / Spring Boot fat-jar),通过命名空间隔离业务域。 AgentScope Java 2.0 原生嵌入 Spring Cloud 体系,通过 Nacos 服务注册与配置中心治理,Higress AI 网关统一接入。 LLM 推理单独占用 GPU 节点,避免与业务服务争抢资源。

    1.1 Nacos + Higress 服务治理体系(融合 Spring Cloud)

    Automy 集群版以团队既有的 Spring Cloud 体系为底座,通过 Nacos 与 Higress 两个阿里生态组件 补齐服务治理与 AI 网关能力,AgentScope Java 2.0 原生嵌入该体系,无需引入额外桥接层。

    🧭 Nacos — 服务注册 + 配置中心 + Skill/MCP 市场

    • 服务注册与发现:所有 Spring Boot 服务(Session/Skill/Expert/Audit/AgentScope Runtime/MCP Server)启动即注册到 Nacos,通过服务名软寻址,Pod 扩缩容自动同步实例列表
    • 配置中心:动态下发模型路由、限流阈值、审批策略、Skill 开关等配置,@RefreshScope 热更新无需重启
    • Skill / MCP 市场管理:AgentScope Toolkit 与 MCP Server 注册到 Nacos 作为「市场目录」,运维侧统一管控上下架、版本、租户可见性
    • 与 Spring Cloud 融合:复用 spring-cloud-starter-alibaba-nacos-discovery + nacos-config,与团队既有微服务体系无缝对接

    🌐 Higress — AI 网关(模型代理 + MCP 代理)

    • 模型代理:统一接入 vLLM / DashScope / 商用 API,屏蔽后端模型差异,按 expert_id / skill_id 路由到推荐模型
    • MCP 代理:对外暴露统一 MCP 入口,后端 MCP Server 经 Nacos 发现并转发,工具调用统一鉴权与审计
    • 统一鉴权:JWT / OIDC(Keycloak)对接,Token 校验 + 多租户隔离 + 工具调用审批联动 AgentScope Permission
    • 限流与审计:按用户/部门/模型维度限流;每次模型与 MCP 调用落审计日志,满足金融合规
    • 与 Spring Cloud 融合:Higress 从 Nacos 拉取服务实例做动态路由,无需硬编码后端地址;替代原 Kong 网关

    3.1 单机部署(开发/测试)

    试用版基于 Spring Boot fat-jar + Docker Compose 一键拉起,基础镜像统一采用 eclipse-temurin:17-jre, AgentScope Java 2.0 以 Maven 依赖嵌入 fat-jar,无需独立 Go 工具链。

    docker-compose.yml(单机版 · Spring Boot fat-jar) 
    # 单机试用版:Spring Boot fat-jar + Docker Compose
# 应用镜像基镜:eclipse-temurin:17-jre(JDK 17+),Maven 构建产出 fat-jar
version: '3.8'
services:
  frontend:
    image: automy/frontend:latest
    ports: ["80:80"]
  api-gateway:
    image: automy/api-gateway:latest  # Spring Boot fat-jar (Higress 单机可省)
    ports: ["8000:8000"]
    depends_on: [postgres, milvus, redis]
  agent-runtime:
    image: automy/agent-runtime:latest  # AgentScope Java 2.0 + Spring Boot fat-jar
    environment:
      MODEL_ENDPOINT: http://vllm:8000
      REDIS_HOST: redis
      MINIO_ENDPOINT: http://minio:9000
      NATS_URL: nats://nats:4222
  postgres:
    image: postgres:15
    volumes: ["pgdata:/var/lib/postgresql/data"]
  milvus:
    image: milvusdb/milvus:v2.4.0
  redis:
    image: redis:7-alpine  # AgentScope Redis Session

  pgdata: {}

    3.2 集群部署(生产)

    生产集群基于 K8s + Spring Cloud + Nacos + Higress 构建高可用体系: 所有 Java 服务以 Spring Boot fat-jar 打包,基镜 eclipse-temurin:17-jre,通过 Maven 多阶段构建产出镜像, 注册到 Nacos 后由 Higress AI 网关统一接入与治理。

    Dockerfile(Spring Boot fat-jar · 多阶段 Maven 构建) 
    # 1. Maven 构建阶段
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests  # 产出 spring-boot fat-jar

# 2. 运行阶段(JDK 17+ JRE)
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar
ENV JAVA_OPTS="-XX:+UseG1GC -XX:MaxRAMPercentage=75"
EXPOSE 8080
ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
    • 高可用:每个服务至少 3 副本,跨节点反亲和部署
    • 自动扩缩容:HPA 基于 CPU/QPS 自动扩缩容,AgentScope Runtime 5-20 副本弹性
    • 滚动更新:Deployment RollingUpdate 策略,maxSurge=1 maxUnavailable=0,零停机
    • 金丝雀发布:Higress 网关流量切分 + Argo Rollouts 实现 10% → 50% → 100% 切换
    • Pod 健康检查:liveness/readiness probe(Actuator /health),失败自动重启/摘除
    • 资源配额:每个 namespace 设置 ResourceQuota,防止资源争抢
    • 服务治理:Spring Cloud + Nacos 服务注册/配置中心,Higress 统一限流/鉴权/审计

    3.3 离线部署(金融内网)

    投资公司核心内网通常完全隔离外网,需全离线部署方案:模型与公共 Skill/MCP 镜像离线导入,MCP 服务全部内网部署。

    • 私有镜像仓库:Harbor 部署在内网,所有 Java/Spring Boot 镜像离线导入(含 eclipse-temurin:17-jre 基镜)
    • Helm Chart 离线包:打包所有依赖 Chart(含 Nacos / Higress / AgentScope Runtime),无外网拉取
    • 模型文件预置:LLM 权重文件(Qwen/DeepSeek/GLM)提前下载,通过 PV 挂载到 vLLM 推理节点
    • 公共 Skill / MCP 镜像离线导入:AgentScope 公共 Skill 与 MCP Server 镜像打包后通过安全通道导入 Harbor,Nacos 中维护离线 Skill/MCP 市场目录
    • MCP 服务内网部署:所有 MCP Server(CRM/ERP/Wind/工商等)仅在内网暴露,通过 Nacos 服务发现,Higress MCP 代理统一鉴权
    • 文档/帮助本地化:所有文档与依赖 CDN 资源本地化
    • 升级包:通过 U 盘/安全传输通道传入内网,版本化升级

    3.4 AgentScope HarnessAgent K8s Deployment

    AgentScope HarnessAgent 服务是智能体层的核心有状态负载,需特别设计 K8s Deployment 以支撑企业级长任务的 跨副本会话恢复、Workspace 快照持久化、零停机滚动发布、多副本水平扩展。状态外置到 Redis(分布式 Session)与 MinIO(Workspace 快照),Pod 本身保持无状态,从而支持任意副本数横向扩缩。

    Deployment 维度配置说明
    镜像 eclipse-temurin:17-jre + Spring Boot fat-jar Maven 多阶段构建,内嵌 AgentScope Java 2.0 依赖
    副本数 5(HPA 弹性 5-20) 基于 CPU/QPS 自动扩缩,跨节点反亲和 podAntiAffinity
    Redis Session RedisSession(jedisPool) 会话上下文外置到 Redis 哨兵集群,任意 Pod 副本均可 resume(sessionId) 恢复,跨副本会话零感知
    Workspace 快照 MinioFilesystemSpec + 定时快照 Workspace 按 sessionId/userId 隔离,快照持久化到 MinIO 对象存储;Pod 重调度后从 MinIO 拉取快照恢复
    滚动发布 maxSurge=1, maxUnavailable=0 零停机:新 Pod readiness 通过后才摘除旧 Pod,Session 已外置不影响在途请求
    健康检查 liveness + readiness(Actuator) /actuator/health/liveness 与 /readiness,AgentScope 运行时就绪后才接流量
    优雅下线 terminationGracePeriodSeconds=60 PreStop hook 刷新 MEMORY.md 与 Workspace 快照到 MinIO,再 deregister Nacos,确保在途任务可恢复
    水平扩展 HPA + ClusterIP round-robin 状态全外置(Redis + MinIO),副本数可线性扩展,无单点瓶颈
    💡
    零停机滚动发布的关键
    HarnessAgent 把 Session 外置到 Redis、Workspace 快照外置到 MinIO,Pod 本身无状态。滚动发布时旧 Pod 在 PreStop 阶段把内存中的 MEMORY.md 与 Workspace 增量刷盘,新 Pod 启动后从 Redis/MinIO 恢复上下文,在途长任务可通过 resume(sessionId) 在新副本续接,实现真正零停机。

    5.3 LangFuse / ARMS 接入说明

    Agent Trace 可观测是 Agent 平台区别于普通微服务的核心能力。Automy 在 OTel Collector 统一采集之上, 对接 LangFuse(开源)与阿里云 ARMS(商业)两个 Agent Trace 平台,按部署形态灵活选择:

    🔍 LangFuse(开源 Agent Trace 平台)

    • 定位:开源 LLM Agent 可观测平台,自托管,Apache-2.0 友好
    • 适用形态:私有化单机版 / 集群版 / 全离线部署(金融内网自托管,数据不出网)
    • 接入方式:OTel Collector 通过 OTLP 导出 → LangFuse;或 AgentScope SDK 直接对接 LangFuse SDK
    • 核心能力:Agent 调用链 Span 树、Token 用量统计、Prompt/Response 审计、用户反馈关联、成本分析
    • 部署:Docker Compose / K8s Helm Chart,内嵌 PostgreSQL 存储

    📊 阿里云 ARMS(商业 Agent 可观测)

    • 定位阿里云应用实时监控服务,商业级 Agent 可观测产品,支持 LLM/AI 应用监控
    • 适用形态:公有云部署或混合云(数据可出网的场景),免运维
    • 接入方式:OTel Collector 导出到 ARMS Endpoint;或 Java Agent 字节码增强(无侵入)
    • 核心能力:端到端 Trace、AI 调用拓扑、模型推理耗时分布、异常根因分析、告警集成
    • 选型建议:对数据不出网无硬性要求时优先 ARMS,省自运维成本;金融内网必须 LangFuse 自托管
    接入环节说明组件
    自动埋点 AgentScope 每个 Reasoning step / Tool call / Model generate 自动生成 Span,带 session_id / user_id / expert_id / skill_id 等业务属性 AgentScope 内置 OTel
    统一采集 所有 Java 服务(Spring Boot + AgentScope)通过 OTLP gRPC/HTTP 上报到 OTel Collector,统一处理、聚合、脱敏 OpenTelemetry Collector
    Trace 路由 Collector 按 span 属性路由:通用基础设施 Trace → Jaeger/Tempo;Agent 业务 Trace → LangFuse / ARMS OTel Collector pipeline
    Agent Trace 存储 LangFuse 自托管(内嵌 PG)或 ARMS(云端);保留期按合规要求设置(默认 90 天,金融场景 1 年) LangFuse / ARMS
    可视化与告警 Span 树可视化、慢节点定位、Token 用量、错误率告警;Trace 与审计日志通过 trace_id 关联 LangFuse / ARMS / Grafana
    📡
    选型建议:LangFuse vs ARMS
    私有化/全离线(金融内网)场景必须选 LangFuse 自托管,确保 Agent Trace 数据不出网;公有云或混合云且无数据出网硬约束的场景可选 ARMS,省去自运维成本。两者均通过 OTel Collector 统一接入,切换仅需改 Collector 导出配置,业务代码零改动。

    七、升级与回滚

    升级流程(零停机)
    1. 版本准备:CI/CD 构建新版本镜像,推送到 Harbor
    2. 测试环境验证:在测试环境完整验证(功能 + 性能 + 安全)
    3. 数据库迁移:执行 schema migration(向后兼容,先迁移再发布)
    4. 金丝雀发布:Argo Rollouts 切 10% 流量到新版本
    5. 监控观察:观察 30 分钟,对比核心指标(错误率、延迟、业务量)
    6. 分批发布:10% → 50% → 100%,每批观察 15 分钟
    7. 全量完成:100% 流量切换,保留旧版本 24 小时用于回滚
    回滚预案
    • 自动回滚:监控触发(错误率 > 5% 或 P99 > 10s)自动切回
    • 手动回滚:`kubectl rollout undo deployment/xxx --to-revision=N`
    • 数据库回滚:migration 提供反向脚本,或 PITR 恢复
    • 配置回滚:ConfigMap 版本化,GitOps 一键回滚
    • 模型回滚:LLM 模型版本切换,保留旧权重
    • 回滚演练:每季度进行回滚演练,验证可行性

    九、容量规划与弹性

    • HPA 水平扩缩容:基于 CPU/QPS 自动调整副本数,AgentScope Runtime 5-20 副本弹性
    • VPA 垂直扩缩容:基于内存自动调整 Pod 资源
    • GPU 弹性:vLLM 推理实例根据队列长度动态扩缩,闲时缩容节省 GPU
    • 分级存储:热数据 PG/SSD,冷数据 S3/机械盘,降低成本
    • 推理批处理:低峰期合并请求,提升 GPU 利用率
    • 缓存预热:工作时间前预热热点知识缓存,降低首屏延迟
