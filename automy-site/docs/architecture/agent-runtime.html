    • 智能体运行时(Eino ADK 专题)

    智能体运行时是 Automy 的核心引擎,基于字节跳动开源的 Eino ADK(Agent Development Kit)构建。 本章节系统性地介绍 Eino ADK 的设计哲学、核心组件、在 Automy 中的落地实践、与技能/专家层的关系、 模型网关对接、选型对比、局限对策,以及基于 OpenTelemetry 的执行追踪设计。这是技术架构中最核心的章节。

    ⚡
    Eino ADK 一句话定义
    字节跳动开源的 Go 原生 LLM 应用框架,组件化拼装 + 声明式 Chain/Graph 编排,企业级高性能智能体开发首选。

    一、Eino ADK 简介

    Eino(Elastic AI Orchestration)是字节跳动于 2024 年开源的 Go 语言 LLM 应用开发框架,定位对标 LangChain/LangGraph,但专为生产级企业场景设计。其 ADK(Agent Development Kit)层提供完整的 Agent 编排能力。

    核心特性

    🧩 组件化拼装

    ChatModel / Tool / Memory / Retriever / Loader 等标准化接口,组件可独立实现与替换,生态兼容性强。

    🔀 声明式编排

    通过 Chain(线性)与 Graph(DAG)两种编排原语,声明式描述工作流,支持分支、并行、循环、条件跳转。

    🚀 高性能

    Go 原生实现,Goroutine 并发模型,流式处理贯穿全链路。无 GIL 限制,单实例支撑高并发。

    🔗 流式优先

    所有组件接口默认支持流式输入输出(Stream API),LLM Token 可端到端流式回传。

    🛡 类型安全

    基于 Go 泛型与接口契约,编译期类型检查,避免运行时错误。

    📊 可观测内置

    原生集成 OpenTelemetry,每一步 Graph 节点自动生成 Span,无需侵入式埋点。

    开源生态

    • 官方仓库:github.com/cloudwego/eino(主体框架)、github.com/cloudwego/eino-ext(扩展组件)
    • 维护方:字节跳动 CloudWeGo 团队(同 Hertz / Kitex 框架)
    • 生态组件:OpenAI / Claude / 通义千问 / DeepSeek / Ollama 等模型适配;Elasticsearch / Milvus / Volc VesDB 等检索后端
    • 许可证:Apache-2.0,商用友好

    1. Chain — 线性编排

    Chain 是最简单的编排原语,按顺序串联多个节点,前一个节点的输出作为后一个节点的输入。适合单一职责的线性流程,如「加载文档 → 分块 → 向量化 → 入库」。

    // Eino Chain 示例:文档索引流水线
chain := NewChain[]*Document]()
    .Append(loader.NewMinIOLoader(bucket))
    .Append(transformer.NewSemanticSplitter(512, 64))
    .Append(encoder.NewBGE_M3(endpoint))
    .Append(retriever.NewMilvusIndexer(collection))

result, err := chain.Invoke(ctx, docs)

    2. Graph — DAG 图编排

    Graph 是 Eino ADK 最强大的编排原语,支持有向无环图(DAG)建模,可表达分支、并行、合并、循环(ReAct 模式)。每个节点(Node)是独立的处理单元,边(Edge)定义数据流向与条件跳转。

    能力说明Automy 应用场景
    分支(Branch)按条件选择下游节点意图识别后路由到不同专家子图
    并行(Parallel)多个节点同时执行财报解析 + 法律扫描 + 行业研究并行
    合并(Merge)多路输出汇合并行结果聚合后生成综合报告
    循环(Loop)带终止条件的迭代ReAct 多轮工具调用、报告迭代优化
    子图(SubGraph)Graph 嵌套 Graph技能复用:把技能作为子图嵌入专家工作流
    状态(State)全局上下文与中间变量跨节点共享会话上下文、记忆、工具结果
     
    // Eino Graph 示例:尽调报告工作流
g := NewGraph().
    AddNode("extract", extractMetricsNode).       // 提取财报指标
    AddNode("legal", legalScanNode).             // 法律风险扫描
    AddNode("industry", industryResearchNode).     // 行业研究
    AddNode("merge", mergeFindingsNode).           // 合并发现
    AddNode("report", generateReportNode).         // 生成报告
    AddEdge("extract", "merge").
    AddEdge("legal", "merge").
    AddEdge("industry", "merge").       // 三路并行 → 合并
    AddEdge("merge", "report")

// 并行执行 extract/legal/industry,完成后触发 merge
err := g.Compile().Invoke(ctx, input)

    3. Tool — 工具抽象

    Tool 是 Agent 调用外部能力的标准接口,基于 Function Calling 模式。Automy 通过 Tool 接口封装 MCP 协议,使所有 MCP Server 提供的工具都能被 Agent 直接调用。

    • ToolsNode:批量工具调用节点,接入 LLM 的 Function Calling 决策
    • ToolInput / ToolOutput:标准化输入输出,带 JSON Schema 描述
    • InvokableTool:无状态工具(查询类)
    • StreamableTool:流式工具(如长任务进度推送)

    4. Memory — 记忆抽象

    Memory 抽象提供短期会话历史与长期跨会话记忆的统一接口。Automy 在此基础上实现三层记忆架构(详见 记忆系统章节)。

    • ChatHistory:会话消息历史,自动窗口管理(滑动窗口 / 摘要压缩)
    • LongTermMemory:跨会话事实/偏好/事件读写
    • Manager:统一管理短期与长期记忆,Agent 通过 Manager 透明访问

    5. ChatModel — 模型抽象

    6. Retriever — 检索抽象

    Retriever 抽象 RAG 检索能力,Automy 实现了混合检索器(向量 + BM25 关键词 + 元数据过滤),并通过 Reranker 精排提升精度。

    技能 = ADK Graph 的可复用编排片段

    技能是 Automy 中可复用、可组合的能力单元,在 ADK 层面对应一个 SubGraph 或 Chain。技能可以独立开发、测试、发布,然后被多个专家 Graph 嵌入复用。

    • 简单技能:线性流程,如「财务报表解析」= 加载 PDF → 提取表格 → 结构化输出(Chain)
    • 复杂技能:DAG 工作流,如「尽调报告」= 多路并行分析 + 合并 + 报告生成(Graph)
    • 技能组合:专家 Graph 可嵌入多个技能 SubGraph,通过 Graph 嵌套实现能力组合

    专家 = ADK Graph + Persona + 绑定资源

    专家是面向特定领域的高级 Agent,在 ADK 层面对应一个完整的 Graph,附加 Persona(角色设定)与资源绑定(工具/知识/记忆)。

    六、Eino ADK vs LangGraph 选型对比

    在选型阶段,Automy 团队重点对比了 Eino ADK 与 LangGraph(LangChain 旗下),最终选择 Eino ADK。下表是详细对比:

    开发语言 Go Python
    性能 高并发、低内存,Goroutine 模型无 GIL 受 GIL 限制,高并发需多进程,内存占用高
    生态成熟度 较新(2024 开源),中文社区活跃 成熟,英文社区庞大,集成最多
    编排模型 Chain + Graph 声明式,支持子图嵌套 StateGraph 状态机式,节点间显式传递 State

    可观测性 内置 OpenTelemetry,自动 Span 需集成 LangSmith,商业产品
    多 Agent 协同 Agent 抽象 + Graph 嵌套 Multi-Agent 模式成熟(Supervisor 等)
    工具生态 Eino-ext 扩展库,主流模型/向量库支持 LangChain Tools 数百个,生态最丰富
    部署运维 单二进制,容器化简单 Python 依赖管理复杂,虚拟环境/Conda

    1. Go 技术栈统一:Automy 后端统一 Go,Eino 避免引入 Python 桥接的开销与运维复杂度
    2. 高性能:Goroutine 并发模型支撑高负载 Agent 调度,单实例万级 QPS
    3. 企业级稳定:字节内部大规模验证,代码质量与文档完整度高
    4. 国产可控:符合金融行业国产化要求,中文社区支持好
    5. 流式优先:原生 Stream API 与 Automy SSE 流式输出契合

    七、局限与对策

    Eino ADK 作为较新的框架,存在一些生态与能力上的局限,Automy 通过以下对策补齐:

    局限影响对策
    工具生态较新 部分企业系统(如特定 CRM/ERP)无官方 Eino Tool 适配 用 ADK 官方 Tool 接口自研 Adapter,通过 MCP 协议桥接已有企业 MCP Server
    Python 工具桥接 部分 AI 能力(如 Unstructured 解析、特定模型)仅有 Python SDK 用 gRPC/HTTP 调用 Python 微服务(PDF 解析 Worker、Embedding 服务),保持核心编排层纯 Go
    RAG 高级模式 复杂 RAG 模式(HyDE、Multi-Query)需自行实现 在 Retriever 抽象上自研 QueryRewriter、HyDE 生成器、多路召回融合器
    调试工具 可视化调试工具不如 LangSmith 完善 基于 OpenTelemetry + Jaeger 自建 Trace 可视化,Graph 执行轨迹可观测
    示例与教程 中文教程较多但英文资料少 内部维护最佳实践文档 + 模板技能,降低开发门槛

    十、性能优化策略

    关键优化点
    • 流式输出:LLM 生成即返回,降低 TTFT(首字节时间)< 800ms
    • 并行执行:Graph 中无依赖节点自动并行(如法律扫描 + 行业研究 + 财报解析)
    • 工具缓存:相同输入的工具调用结果缓存(Redis,5 分钟 TTL)
    • 记忆预加载:会话开始时预加载相关记忆,避免推理时阻塞
    • 上下文压缩:长会话自动摘要历史,降低 Token 成本(超过 8K Token 触发压缩)
    • 模型分级:简单任务(意图识别、摘要)走小模型,复杂推理走大模型
    • 连接池复用:MCP Client 连接池复用,避免每次调用重建连接
    • Goroutine 池:限制并发 Goroutine 数量,避免 OOM

    十一、执行流程总结

    1. 请求接收:Session Service 接收用户请求,创建会话上下文
    2. 专家加载:根据 expert_id 加载 Expert Graph 定义 + Persona + 资源绑定
    3. 上下文组装:加载会话历史、用户偏好、项目记忆
    4. Graph 编译:Eino ADK 编译 Graph,生成可执行计划
    5. 节点执行:按 Graph 拓扑顺序执行,自动并行化无依赖节点
    6. 工具调用:ToolsNode 接收 LLM 决策,并行调用 MCP 工具
    7. 模型推理:ChatModel 流式调用模型网关,Token 端到端回流
    8. 记忆写入:异步提取本次会话关键信息写入长期记忆
    9. 结果返回:流式 SSE 输出到前端,标注引用与来源
    10. 审计记录:全程 Trace + 事件投递 NATS,审计服务消费落库
