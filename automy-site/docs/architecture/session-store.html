    • 会话数据库设计

    会话数据是 Automy 多端协同与主动提醒的数据底座。本章节定义 Session、Message、SessionEvent、SessionParticipant、PushRecord 五张核心表及其关系,呈现 ER 图、存储分布、索引策略、与记忆系统关系、归档与冷存策略、数据生命周期与合规保留等关键设计, 服务于工作台、IM 通道、记忆系统、主动提醒引擎的协同运转。

    🗄
    设计原则
    热数据 PostgreSQL + 冷数据归档 S3 + 向量库 Milvus 检索 + Redis 在线缓存,冷热分层 · 多端实时同步 · 涉密会话仅站内信 · 全链路审计可追溯

    一、概述

    Automy 会话数据需要承载多端(飞书 / 企微 / 钉钉 / Web / 桌面)发起的会话及其消息流,同时支撑跨端续接、长期归档、与记忆系统联动、主动提醒触达记录等多维场景。会话数据按访问热度分层存储,既保证活跃会话的低延迟访问,又控制长期数据的存储成本。所有数据写入均经过涉密等级判定,涉密会话仅站内信可见,IM 通道不投递原文。

    表用途预估量级分区策略
    Session会话主信息单用户 1000 条,全量百万级不分区,按 user_id 索引
    Message消息内容单会话 100+ 条,全量亿级按月 RANGE 分区
    SessionEvent生命周期事件单会话 10 条,全量千万级不分区,按 session_id 索引
    SessionParticipant参与方单会话 5 条,全量百万级不分区,按 session_id 索引
    PushRecord推送记录单用户日 20 条,全量亿级按月 RANGE 分区

    三、存储分布

    会话数据按访问模式分布在四类存储引擎,通过应用层 Session Service 统一编排,实现冷热分层、检索加速与在线缓存。

    存储引擎承载内容访问模式选型理由
    PostgreSQL 15+活跃会话、消息、事件、参与方读写均衡、事务一致JSONB 支持 tool_calls,分区表支持亿级消息
    S3 / MinIO归档会话、归档消息批量读、低频写对象存储低成本长期留存,跨区域复制
    Milvus消息向量、语义检索高 QPS 向量查询分布式十亿级向量,HNSW 索引低延迟
    Redis 7.x在线会话上下文、最近消息亚毫秒读低延迟、数据结构丰富、TTL 自动过期

    四、索引策略

    针对投资公司高频查询场景,设计四类核心索引,覆盖用户视角的"我的会话"、会话视角的消息拉取、跨端来源追溯、归档扫描等关键路径。

    索引名称字段组合适用查询示例
    idx_sessions_user_time(user_id, last_active_at DESC)用户视角会话列表"张三最近的会话"
    idx_messages_session_seq(session_id, created_at)会话内消息拉取"加载会话最新 20 条消息"
    idx_messages_source_channel(source_channel, created_at)跨端来源追溯"飞书发起的会话消息"
    idx_sessions_status_active(status, last_active_at)归档扫描"扫描 90 天未活跃会话归档"
    idx_push_records_user(user_id, sent_at DESC)通知中心查询"张三最近通知"
    idx_events_session_type(session_id, event_type)生命周期审计"该会话的所有 handed_off 事件"
    PostgreSQL 索引 DDL 示例 
    -- 用户视角会话列表索引
CREATE INDEX idx_sessions_user_time ON sessions(user_id, last_active_at DESC);

-- 会话内消息拉取索引(分区表)
CREATE INDEX idx_messages_session_seq ON messages(session_id, created_at);

-- 跨端来源追溯索引
CREATE INDEX idx_messages_source_channel ON messages(source_channel, created_at);

-- 归档扫描索引
CREATE INDEX idx_sessions_status_active ON sessions(status, last_active_at)
  WHERE status IN ('active', 'paused');

    五、与记忆系统关系

    会话数据是记忆系统的输入源。每条会话消息经记忆提取流水线处理后,沉淀到长期记忆库的事实 / 偏好 / 事件三类记忆,跨会话召回时反向注入到新会话上下文。

    六、归档与冷存策略

    长期未活跃会话自动归档到冷存储,降低热存储压力;归档会话可按需恢复继续使用,恢复后重新进入活跃状态。

    会话归档状态机 Mermaid
    stateDiagram-v2 [*] --> 活跃: 创建会话 活跃 --> 暂停: 用户主动暂停 暂停 --> 活跃: 用户恢复 活跃 --> 归档: 90 天未活跃 暂停 --> 归档: 90 天未活跃 归档 --> 活跃: 用户主动恢复 归档 --> 销毁: 2 年保留到期 note right of 归档 S3 冷存储 元数据保留索引 end note note right of 销毁 合规审计后销毁 日志保留 5 年 end note
    状态触发条件存储位置可访问性
    active新建或最近活跃PostgreSQL + Redis实时读写
    paused用户主动暂停PostgreSQL可读,不触发提醒
    archived90 天未活跃S3 + 元数据索引需恢复后才可读写
    销毁归档 2 年到期仅审计日志不可访问

    七、数据生命周期与合规保留

    数据类型热存储温存储冷归档销毁
    会话 Session0-90 天(PG)30-90 天未活跃(PG)90 天+ 未活跃(S3)归档 2 年后
    消息 Message0-90 天(PG 分区)30-90 天(PG)90 天+(S3)随会话销毁
    SessionEvent永久(PG)--不销毁(审计)
    PushRecord0-90 天(PG 分区)30-90 天(PG)90 天+(S3)归档 2 年后
    涉密会话(L3+)0-90 天(PG 加密)不外迁不归档到 S3合规审计后销毁
    审计日志0-1 年(PG audit 库)1-3 年(S3)3-5 年(S3 WORM)5 年(合规)
    🔒
    涉密会话特殊处理
    L3 及以上涉密等级的会话不归档到对象存储,仅在 PostgreSQL 加密表内沉淀;消息内容仅站内信可见,IM 通道不投递原文;销毁前须经合规审计员审批。
