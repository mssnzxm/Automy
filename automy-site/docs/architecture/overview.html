    • 架构总览

    Automy 奥托迈是面向投资公司的企业级智能体平台,采用分层解耦、标准协议、私有可控的架构理念。 本章节是技术架构的入口,系统性地介绍整体设计原则、五层架构全景、技术栈选型与部署形态, 为后续各章节(系统设计、Eino ADK 运行时、记忆系统、知识库、MCP 协议等)提供宏观视角。

    🏛
    架构定位
    面向投资公司内部使用的私有化智能体平台 · 数据主权归企业所有 · 全链路审计可追溯

    一、设计原则

    Automy 在架构设计上遵循以下五大原则,这些原则贯穿从基础设施到用户体验的每一层,确保系统在企业级场景下的可靠性、安全性与可演进性。

    1. 分层解耦

    五层架构(用户体验 / 应用服务 / 智能体核心 / 数据模型 / 基础设施)各司其职,通过标准接口交互。任一层可独立演进、替换实现,例如将向量库从 Milvus 切换到 Qdrant 不影响上层 Agent 运行时。

    2. 标准协议

    对外采用开放标准:模型调用走 OpenAI 兼容 API、企业系统接入走 MCP 协议、模型推理走 OpenTelemetry。避免厂商锁定,生态可持续扩展。

    3. 私有可控

    数据全链路不出企业边界:本地推理(Qwen / DeepSeek / GLM)、本地存储(PostgreSQL / Milvus / MinIO)、本地认证(Keycloak)。满足金融行业合规与等保要求。

    4. 可观测

    从 LLM Token 用量到 Agent 工作流每一步,全链路埋点。Prometheus 指标 + Loki 日志 + OpenTelemetry Trace 三位一体,支撑性能调优与故障定位。

    5. 可演进

    模块化设计 + 版本化 API + 技能/专家热更新。新模型、新工具、新业务场景可通过市场机制或灰度发布上线,无需平台停机升级。

    6. 默认安全

    最小权限原则(RBAC + ABAC)、纵深防御(网络/容器/数据多层隔离)、敏感数据脱敏与全链路审计。安全不是附加项,而是默认配置。

    Automy 的技术选型遵循「成熟开源 + 国产可控 + 企业级验证」三大准则。下表列出各层关键技术与选型理由。

    层级技术说明
    前端 WebReact + Vite + Tailwind + shadcn/ui组件化开发,Vite 极速构建,shadcn/ui 提供企业级 UI
    桌面端Electron跨平台桌面应用,支持 Windows / macOS / Linux
    后端服务Go (Gin / Echo)高性能、低内存,适合微服务与高并发场景

    关系数据库PostgreSQL 15+JSONB、分区表、pgvector 扩展
    对象存储MinIOS3 兼容,私有化部署,文档/附件/备份

    四、核心架构特征

    ⚡ 高性能

    • Go 后端,单机万级 QPS
    • LLM 流式输出,TTFT < 800ms
    • 无依赖步骤并行执行
    • 工具调用结果缓存

    📈 可扩展

    • 微服务无状态水平扩展
    • Milvus 分布式向量检索
    • 模型推理 GPU 弹性扩缩
    • 技能/专家热加载

    🔒 高安全

    • 全链路 TLS 1.3 加密
    • AES-256 静态加密
    • RBAC + ABAC 双重权限
    • 审计日志签名链防篡改

    五、部署形态

    Automy 提供三种部署形态,适配不同规模与场景的投资公司:

    🖥 单机部署

    开发 / POC

    • Docker Compose 一键启动
    • 适合 50 人以下团队试用
    • 1×A100 GPU 即可运行
    • 组件全部容器化,资源占用低

    ☁ 集群部署

    生产推荐

    • Kubernetes 编排,高可用
    • 支持 200-1000+ 用户
    • 多 GPU 推理池
    • 滚动更新与金丝雀发布

    🔒 离线部署

    金融隔离网络

    • 完整镜像离线包
    • 无任何外网依赖
    • 支持物理机与私有云
    • 符合金融行业隔离要求
    💡
    金融场景适配
    投资公司通常有严格的网络隔离要求,Automy 离线部署形态可完全运行在物理隔离内网中,模型、依赖、镜像全部预打包,安装过程不依赖任何外网。

    六、关键设计决策

    1. 为什么选择 Eino ADK 而非 LangChain / LangGraph?

    Eino 是字节跳动开源的 LLM 应用框架,选择它的核心理由:

    • Go 原生:与 Automy 后端技术栈统一,避免 Python 桥接开销,Goroutine 并发模型适合高负载 Agent 调度
    • 声明式编排:Chain/Graph 抽象清晰,支持分支、并行、循环,适合复杂业务工作流
    • 组件化拼装:ChatModel / Tool / Memory / Retriever 标准接口,组件可独立替换
    • 企业级稳定:字节内部大规模验证,代码质量与文档完整度高
    • 国产可控:符合国产化要求,中文社区支持好

    详细对比见 智能体运行时章节。

    2. 为什么采用 MCP 协议接入企业系统?

    MCP(Model Context Protocol)是 Anthropic 主导的开放标准,优势:

    • 标准化:统一 LLM 与外部系统交互方式,避免每个集成重复造轮子
    • 生态丰富:越来越多企业系统提供 MCP 服务,可直接复用
    • 解耦:Agent 与企业系统解耦,可独立演进、独立升级
    • 可控:权限模型清晰,Resource/Tool/Prompt 三类能力精确授权,适合企业场景
    • 可观测:每次调用都有结构化日志与审计记录
    3. 为什么强调私有化部署?

    投资公司数据高度敏感(客户清单、交易数据、研究观点、未公开项目信息),私有化部署是底线要求:

    • 数据不出企业边界:所有数据(会话、记忆、文档)在企业内网流转
    • 本地推理:对接 vLLM/TGI 自部署 Qwen/DeepSeek/GLM,无外部 API 调用
    • 对接企业 SSO:与现有身份系统无缝集成,不引入新的账号体系
    • 合规审计:全链路审计日志留存企业本地,满足监管检查
    • 性能可控:不受公网延迟与第三方限流影响
    4. 为什么用 PostgreSQL 而非 MySQL?
    • JSONB 支持:记忆/技能/审计日志大量使用 JSONB,索引与查询性能优于 MySQL JSON
    • pgvector 扩展:可作为向量库的备份/小型场景方案,与 Milvus 互补
    • 分区表:消息表按月分区,千万级消息查询性能稳定
    • 扩展生态:pgvector、pg_repack、pglogical 等扩展丰富
    5. 为什么用 NATS 而非 Kafka?
    • 轻量:NATS 单二进制部署,资源占用远低于 Kafka,适合私有化场景
    • 高性能:百万级消息/秒,延迟亚毫秒级
    • JetStream:支持持久化、At-Least-Once 投递,满足审计与可靠性要求
    • 运维简单:无需 Zookeeper / KRaas,部署运维成本低

    七、架构权衡与边界

    权衡点选择取舍说明
    性能 vs 灵活性 偏向灵活性 Eino ADK Graph 声明式编排带来运行时解析开销,但换取了技能/专家的可视化编排与动态加载能力
    一致性 vs 可用性 最终一致性 记忆写入异步化,换取低延迟响应;关键审计日志走强一致性写入
    通用 vs 专用 领域专用 预置投研/法务/合规专家,牺牲通用性换取金融场景开箱即用
    生态 vs 可控 国产可控优先 选择 Eino ADK 而非 LangChain,生态略小但国产化、Go 技术栈统一
