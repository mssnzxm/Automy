    • 记忆系统设计

    记忆是 Automy 让 AI 成为「长期伙伴」的核心能力。区别于一次性问答机器人,Automy 通过分层记忆架构, 让 Agent 跨会话延续上下文、理解用户偏好、记住项目背景。本章节介绍三层记忆架构、写入流水线、 检索策略与遗忘机制,所有设计面向投资公司高强度业务场景。

    🧠
    设计目标
    让 Agent 记住用户偏好 · 跨会话延续项目上下文 · 主动遗忘过期信息 · 数据主权归企业所有

    一、三层记忆架构

    Automy 记忆系统采用三层架构:短期(Redis)、工作(会话级)、长期(PG + 向量库)。各层职责清晰、生命周期不同,通过 Memory Manager 统一对外提供接口。

    二、记忆数据模型

    Automy 记忆按类型分为三大类:事实(Fact)、偏好(Preference)、事件(Event),每条记忆有结构化字段。

    类型说明示例
    事实 Fact客观信息,可被引用与验证「ABC 公司 2025 年营收 12 亿元」
    偏好 Preference用户主观偏好或工作习惯「该用户偏好简洁的要点式回答」
    事件 Event带时间戳的事件记录「2026-07-10 用户决定下投 ABC 公司 A 轮」
    PostgreSQL Schema - memories 表 
    CREATE TABLE memories (
  id VARCHAR(36) PRIMARY KEY,
  user_id VARCHAR(36) NOT NULL,
  user_name VARCHAR(64),
  type VARCHAR(32) NOT NULL,  -- fact / preference / event
  category VARCHAR(64),         -- 行业分类:投研/法务/合规
  content TEXT NOT NULL,         -- 记忆内容
  scope VARCHAR(32) NOT NULL,     -- session / project / user / global
  scope_id VARCHAR(36),           -- 项目 ID(项目级记忆)
  source VARCHAR(32),            -- explicit / inferred / conversation / meeting
  confidence FLOAT DEFAULT 1.0,  -- 置信度
  tags JSONB,                  -- 标签(行业/公司/部门)
  vector_id VARCHAR(64),          -- Milvus 向量 ID
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  last_accessed TIMESTAMP,        -- 最后访问时间(用于遗忘)
  access_count INT DEFAULT 0,    -- 访问次数
  importance FLOAT DEFAULT 0.5, -- 重要性评分(0-1)
  is_deleted BOOLEAN DEFAULT FALSE
);

CREATE INDEX idx_memories_user ON memories(user_id);
CREATE INDEX idx_memories_scope ON memories(scope, scope_id);
CREATE INDEX idx_memories_tags ON memories USING GIN(tags);
CREATE INDEX idx_memories_type ON memories(user_id, type, is_deleted);

    三、记忆写入流水线

    记忆写入不是简单的 INSERT,而是一条完整的流水线:LLM 提取 → 冲突检测 → 合并/覆盖 → 入库。这条流水线保证记忆的准确性、不冲突、可追溯。

    记忆写入流水线 Mermaid
    flowchart LR TRIGGER["触发\n会话结束/主动触发"] --> EXTRACT["LLM 提取\n结构化记忆条目"] EXTRACT --> CLASSIFY["分类\n判断层级 user/project/global"] CLASSIFY --> CONFLICT{"冲突检测\n与已有记忆比对"} CONFLICT -->|无冲突| EMBED["向量化\nBGE-M3"] CONFLICT -->|有冲突| MERGE["合并/覆盖\n按策略处理"] MERGE --> EMBED EMBED --> STORE_PG[(PostgreSQL)] EMBED --> STORE_VEC[(Milvus)] STORE_PG --> INDEX["更新索引"] STORE_VEC --> INDEX INDEX --> AUDIT["审计日志\n记录写入"] MERGE -->|覆盖旧记忆| ARCHIVE["归档旧版本\n可追溯"]

    流水线各阶段说明

    阶段关键动作技术实现
    1. 触发会话结束、用户主动「记住这个」、定时任务事件总线 NATS + 定时 Cron
    2. LLM 提取用专门 Prompt 让 LLM 从对话中提取事实/偏好/事件Qwen2.5-72B + 结构化输出
    3. 分类判断记忆归属层级(user/project/global)规则 + LLM 混合判定
    4. 冲突检测向量检索相似记忆,判断是否矛盾Milvus Top-1 相似度 > 0.92 视为潜在冲突
    5. 合并/覆盖新记忆覆盖旧的、或合并补充按类型策略:偏好覆盖、事实合并、事件追加
    6. 向量化生成 BGE-M3 向量本地 Embedding 服务
    7. 入库元数据入 PG,向量入 Milvus双写事务保证一致性
    8. 归档被覆盖的旧记忆归档,可追溯memory_versions 表
    9. 审计记录写入操作到审计日志NATS → Audit Service

    冲突处理策略

    记忆类型冲突策略示例
    偏好 Preference新覆盖旧用户从「喜欢详细回答」改为「喜欢简洁回答」
    事实 Fact合并(新旧都保留,标注时间)公司营收数据更新,旧数据保留为历史
    事件 Event追加(不冲突)不同时间的事件独立保存
    显式纠正强制覆盖 + 标记纠正用户主动「不是这样,应该是...」

    四、记忆检索策略

    Automy 采用「向量检索 + 关键词检索 + 元数据过滤」的混合检索策略,并通过重排序提升精度。

    flowchart LR Q["当前会话上下文\n生成检索 Query"] --> REWRITE["Query 改写\n指代消解 + 扩展"] REWRITE --> VEC["向量检索\nTop-K=10"] REWRITE --> KW["关键词检索\nBM25 Top-K=10"] VEC --> FUSION["融合排序\nRRF 算法"] KW --> FUSION FUSION --> FILTER["元数据过滤\nscope/user_id/tags"] FILTER --> RERANK["重排序\nBGE-Reranker"] RERANK --> TOP["取 Top-5"] TOP --> INJECT["注入 Agent 上下文"]

    检索细节

    环节方法说明
    Query 生成当前会话最近 3 轮 + 用户长期意图避免单轮 Query 信息不足
    Query 改写指代消解 + HyDE 假设文档「他」→ 具体人名,提升召回
    向量检索Milvus HNSW,Top-K=10语义相似召回
    关键词检索BM25,Top-K=10专有名词、代码精确匹配
    融合排序RRF(Reciprocal Rank Fusion)向量与关键词结果融合
    元数据过滤scope / user_id / tags权限隔离,只能看自己的记忆
    重排序BGE-Reranker-v2 Cross-Encoder精排提升 Top-K 精度
    结果数量Top-5 注入上下文避免上下文过长
    引用标注每条记忆带 ID + 时间Agent 回答时可引用记忆来源

    五、遗忘机制

    记忆系统不能只增不减,必须有遗忘机制,否则长期记忆会被低价值信息淹没。Automy 采用三种遗忘策略组合:

    ⏰ 时间衰减

    基于 Ebbinghaus 遗忘曲线,长期未访问的记忆重要性递减。超过 6 个月未访问且 importance < 0.3 的记忆自动归档。

    公式:score = importance × decay(now - last_accessed)

    ✏ 显式纠正

    用户主动纠正的错误记忆,标记为「已纠正」并降低 confidence。同一主题多次纠正的反向旧记忆,归档处理。

    🗑 主动删除

    用户在「记忆管理」界面主动删除,软删除标记 is_deleted,30 天后物理删除。删除前需二次确认,操作审计留存。

    遗忘执行策略

    # 遗忘策略配置
forgetting:
  time_decay:
    enabled: true
    half_life_days: 90           # 90 天半衰期
    archive_threshold: 0.3        # 重要性低于 0.3 归档
    archive_after_days: 180      # 180 天未访问触发评估
  explicit_correction:
    confidence_drop: 0.5          # 纠正后置信度降一半
    archive_after_corrections: 3  # 同主题 3 次纠正归档旧记忆
  active_delete:
    soft_delete_days: 30          # 软删除 30 天后物理删除
    require_confirm: true          # 需二次确认
    audit_retain: 5               # 删除操作审计保留 5 年
  scheduled_task:
    cron: "0 3 * * *"              # 每天凌晨 3 点执行

    八、隐私与合规

    🔒
    记忆数据保护
    记忆数据高度敏感,涉及用户偏好与项目信息,Automy 提供多层保护:
    • 权限隔离:用户只能检索自己的 user 级记忆;project 级记忆按项目成员访问;global 级全员可见
    • 存储加密:记忆内容字段 AES-256 加密存储,密钥由 KMS 管理
    • 导出与删除:用户可随时导出全部记忆(JSON),也可一键清除(软删除 30 天后物理删除)
    • 审计可追溯:每次记忆读/写/删除操作都记录到审计日志,带 trace_id 关联
    • 数据保留期:项目归档后保留 1 年,用户离职后保留 6 个月(可配置),合规要求 5 年
