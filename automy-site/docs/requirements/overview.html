    • 需求文档总览

    Automy 奥托迈是企业级自主智能体平台,面向投资公司内部使用。本文档体系完整描述了 Automy 的 12 个核心需求模块,涵盖从用户体验到底层基础设施的全部能力。每个模块都包含需求背景、功能列表、 交互流程与验收标准。

    📋
    文档结构
    需求文档共 12 个模块,按用户体验、核心能力、平台基础设施三个层次组织。所有需求条目采用统一编号规则 REQ-X.Y,X 为模块号,Y 为模块内需求序号,优先级标注 P0/P1/P2。

    模块概述

    Automy 奥托迈作为投资公司内部的智能体平台,其核心使命是将通用大模型能力转化为可落地、可治理、可演进的企业级生产力工具。本文档体系从三个层次展开:面向最终员工的用户体验层提供自然的工作流接入;面向业务差异化竞争力的核心能力层构建技能、专家、知识与集成生态;面向 IT、合规与运维的平台基础设施层保障数据主权、合规与稳定运行。

    每个模块的需求条目均遵循 RFC 2119 规范,使用 SHALL(必须)、SHOULD(应当)、MAY(可选)三级关键词明示实现强度,并配套触发场景(WHEN / THEN)、关联原型与架构文档,以便产品、研发、测试与审计人员对齐一致理解。

    第一层:用户体验

    面向最终用户(投资公司员工)的交互层需求,聚焦"如何使用 Automy"。这一层目标是降低使用门槛,让投资经理、研究员、法务、合规、风控等角色不切换系统即可获得 AI 协助,并在长期使用中感受到"懂我"的个性化体验。

    阅读建议

    📌 阅读顺序建议
    1. 产品经理 / 业务方:从 工作台与会话 开始,理解用户体验,再依次阅读 长期陪伴与记忆 与 专家市场。
    2. 架构师:从 技术架构总览 开始,理解系统设计,对照本文档的需求编号与架构章节建立映射。
    3. 开发工程师:按模块逐个阅读,重点关注每个 REQ 条目的 SHALL 描述与 WHEN/THEN 场景,作为实现与测试依据。
    4. 运维 / 安全:重点阅读 安全合规 与 管理后台,关注 REQ-9.x 与 REQ-10.x 的合规、审计、KMS、数据驻留要求。
    5. 合规 / 内审:重点关注 REQ-9.3 全链路审计、REQ-9.4 脱敏与合规、REQ-10.3 审计日志查看。

    需求编写规范

    📐 编号规则

    采用 REQ-X.Y 格式,X 为模块编号(1-11),Y 为模块内需求序号。新增需求顺延编号,废弃编号不复用,标注 [Deprecated]。

    🎯 优先级定义

    P0 必须实现,P1 重要可延后,P2 增强性需求。优先级在版本规划会上确定。

    📜 RFC 2119 关键词

    SHALL 表示必须实现,SHOULD 表示应当实现但允许有理由的偏离,MAY 表示可选实现。

    🔗 关联追溯

    每个 REQ 条目需关联对应的原型页与架构章节,支持需求-原型-架构-代码四向追溯。
