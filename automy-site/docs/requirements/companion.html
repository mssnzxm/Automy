    • 长期陪伴与记忆

    通过分层记忆系统让 Automy 真正"记住"每位员工,理解其工作偏好、项目上下文与历史决策,提供个性化、连续性的智能体服务,告别每次对话都要重新介绍背景的低效体验。

    🤝
    模块定位
    本模块定义短期对话记忆、长期记忆库(事实 / 偏好 / 事件三类)、偏好学习、跨会话召回、记忆写入与遗忘、记忆审计等核心能力,让 Agent 成为员工的长期工作伙伴而非一次性问答工具。

    模块概述

    投资公司员工日常工作中,频繁切换不同项目、不同研究主题:上午处理 ABC 新能源尽调,下午看 PQR 消费行业趋势,晚上撰写季度策略报告。传统 LLM 应用每次对话都是孤立的,无法延续上下文,导致每次都要重复说明项目背景、个人偏好、关注条款,既低效又难以形成"懂你"的长期工作关系。

    Automy 通过分层记忆系统解决这一问题:短期记忆承载会话内上下文,长期记忆按"事实 / 偏好 / 事件"三类结构化沉淀,跨会话召回时自动注入,记忆写入与遗忘遵循提取-去重-合并-衰减-审计的完整治理链路,让 AI 既记得住该记的,也忘得了该忘的。

    需求条目

    REQ-2.1 短期对话记忆
    P0

    系统 SHALL 在每个会话内维护短期对话记忆,采用滑动窗口 + 摘要压缩策略,确保长对话上下文不丢失且 Token 成本可控;窗口大小默认 20 轮(可配置),超出后触发早期消息摘要压缩。

    • 滑动窗口:保留最近 N 轮对话原文(默认 20 轮),旧消息进入摘要队列
    • 摘要压缩:每超出 5 轮触发一次摘要,把早期对话压缩为 200 字以内的结构化摘要
    • 关键信息保留:摘要时强制保留实体(人名、公司、金额)、决策结论、未完成任务
    • Token 控制:单次请求上下文 Token 总量不超过模型上限的 70%,超出时优先压缩工具调用细节
    • 会话隔离:不同会话的短期记忆相互独立,关闭会话后释放

    场景

    WHEN 投资经理与专家进行 30 轮对话,讨论 ABC 新能源的财务、技术、市场三个维度
    THEN 系统保留最近 20 轮原文,前 10 轮被压缩为摘要:"已讨论 ABC 新能源 2024 年营收 ¥50 亿(+ 30%),动力电池业务占比 65%,核心技术为磷酸铁锂...";后续提问能正确引用早期结论,无需用户重复。

    REQ-2.2 长期记忆库(事实 / 偏好 / 事件)
    P0

    系统 SHALL 提供长期记忆库,按事实(Fact)、偏好(Preference)、事件(Event)三类结构化存储,每条记忆 SHALL 包含内容、类型、来源、置信度、时间戳、TTL 等元数据;记忆库 SHALL 支持向量检索与结构化检索双通道。

    类型定义示例来源
    Fact 事实客观不变的知识张三任职于投研部,工号 EMP-1024SSO / HR 同步
    Preference 偏好用户的工作风格倾向回复风格偏好简洁、关注新能源赛道、估值默认用 DCF显式设置 / 隐式推断
    Event 事件有明确时间点的发生2026-03-15 参加投决会,通过 XYZ 项目立项对话提取 / 日程同步
    • 记忆 schema:content / type / source / confidence(0-1) / createdAt / updatedAt / TTL / tags
    • 检索通道:语义检索(向量相似度)+ 结构化检索(按类型、标签、时间范围)
    • 项目隔离:项目级记忆与用户级记忆独立存储,跨项目记忆默认不互相干扰
    • 容量限制:单用户记忆条数上限 10000 条,超出按置信度与时间衰减淘汰

    场景

    WHEN 投资经理张三在 3 月 15 日通过对话告知专家:"本周主要看新能源车产业链,重点关注宁德时代和比亚迪"
    THEN 系统提取一条 Preference 记忆:"关注新能源车产业链,重点关注宁德时代、比亚迪",source 标记为会话 #sess-1234,confidence 0.85;后续对话与跨会话时专家自动应用该偏好,优先返回这两家公司的相关信息。

    REQ-2.3 偏好学习
    P0

    系统 SHALL 通过显式设置与隐式推断两种方式持续学习用户偏好,显式偏好来自用户在用户中心的主动配置,隐式偏好来自连续行为沉淀(常用工具、典型查询、关注字段、回复风格),并 SHALL 定期回顾与更新偏好,避免过时偏好干扰决策。

    • 显式偏好:用户在用户中心主动设置关注赛道、回复风格、关注条款、默认参数
    • 隐式偏好:从最近 30 天交互中推断(如频繁查询新能源 → 自动标记为关注赛道)
    • 偏好确认:隐式推断的偏好首次应用时 SHALL 提示用户确认,用户可拒绝
    • 偏好衰减:30 天未被引用的隐式偏好置信度自动衰减,90 天未引用则标记为待清理
    • 偏好冲突:新旧偏好冲突时,以新偏好为准并记录变更原因

    场景

    WHEN 张三连续 5 次与专家对话都要求 "用 DCF 模型估值",并 3 次要求 "回复简短一些,先给结论"
    THEN 系统在 7 天后沉淀一条隐式 Preference:"估值默认使用 DCF 模型;回复风格偏好先结论后展开";首次应用时弹出确认提示,张三确认后该偏好写入长期记忆,后续所有专家对话自动遵循。

    REQ-2.4 跨会话记忆召回
    P0

    系统 SHALL 在新会话开始或用户提问时,自动检索并注入相关记忆作为上下文,无需用户手动 @提及 或重新说明;召回策略 SHALL 综合语义相似度、时间衰减、置信度、项目关联度四维加权排序。

    • 自动注入:用户首条消息发出后,异步检索 Top-K(默认 5)相关记忆注入 system prompt
    • 召回排序:score = 0.4 × 语义相似度 + 0.3 × 时间近度 + 0.2 × 置信度 + 0.1 × 项目关联度
    • 项目优先:当前会话归属某项目时,项目级记忆权重提升 1.5 倍
    • 透明展示:在消息气泡侧边展示 "本次注入了 N 条记忆",用户可点击查看详情
    • 用户可控:用户可在偏好中关闭自动注入,改为手动 @memory 触发

    场景

    WHEN 张三在新会话中提问 "ABC 新能源最新财报有什么变化?"
    THEN 系统检索到 3 条相关记忆:(1) 上周讨论过 ABC 营收 ¥50 亿;(2) 张三关注动力电池业务占比;(3) ABC 是张三的关注标的;自动注入后专家回答:"延续上周讨论,ABC 新能源 2025 Q2 营收 ¥28 亿(环比 +12%),动力电池业务占比从 65% 提升至 70%..."

    REQ-2.5 记忆写入与遗忘
    P1

    系统 SHALL 实现"提取-去重-合并"的记忆写入链路,避免重复记忆污染记忆库;SHALL 支持基于时间衰减与主动遗忘的记忆生命周期管理,过时或低价值记忆自动归档或删除,用户也可主动遗忘指定记忆。

    • 提取:对话结束时由 LLM 提取候选记忆条目,标注类型与置信度
    • 去重:与已有记忆做语义相似度比对,相似度 >0.85 视为重复,触发合并而非新增
    • 合并:重复记忆合并时保留信息更完整、置信度更高、时间更新的版本
    • 时间衰减:90 天未访问的记忆置信度 ×0.8,180 天 ×0.5,365 天自动归档
    • 主动遗忘:用户可 @forget 命令删除指定记忆,或批量清除某项目 / 某时间段的记忆
    • 遗忘审计:所有遗忘操作记录到审计日志,保留 1 年可追溯

    场景

    WHEN 张三在新会话中说 "我最近不再关注 PQR 消费行业了"
    THEN 系统提取一条 Preference 记忆:"不再关注 PQR 消费行业",与已有的 "关注 PQR 消费行业" 做相似度比对,识别为冲突,合并为 "已不再关注 PQR 消费行业(2026-07 更新)",旧记忆置信度降为 0.3 并标记为 archived。
    WHEN 张三执行 @forget project=XYZ-尽调
    THEN 系统列出该项目下所有 12 条记忆,张三确认后批量删除,操作记录到审计日志:"用户主动遗忘 XYZ 项目记忆 12 条",后续召回不再包含这些记忆。

    REQ-2.6 记忆审计
    P1

    系统 SHALL 对记忆的所有读写操作留痕,包括创建、读取、修改、删除、召回注入,审计日志 SHALL 记录操作者、时间、记忆 ID、操作类型、来源会话、变更前后内容摘要;审计日志对合规角色可见,对用户本人可见自己的审计轨迹。

    • 审计字段:operator / operation / memoryId / sessionId / before / after / timestamp
    • 读审计:记忆被召回注入时记录"读取"操作,标记被注入到哪个会话
    • 写审计:记忆创建、修改、删除时记录变更前后内容摘要(完整内容加密存储)
    • 合规视角:auditor 角色可按用户、时间范围查询所有记忆操作,导出合规报告
    • 用户视角:用户可在用户中心查看自己的记忆操作轨迹,质疑异常访问
    • 日志保留:审计日志保留 1 年,敏感操作(删除、批量遗忘)保留 3 年

    场景

    WHEN 合规审计员李四怀疑某员工的记忆被异常修改
    THEN 李四在管理后台 → 审计日志 → 记忆操作,按用户筛选,查看该用户最近 30 天所有记忆写操作,发现 2026-07-10 23:45 有一条管理员修改记录(修改内容:删除了某条项目记忆),可追溯到操作者 admin 账号与操作原因 "员工离职数据清理"。

    关键交互流程

    记忆自动写入流程
    1. 用户与 Agent 进行对话
    2. Agent 识别关键信息(决策、偏好、事实)
    3. 系统提示"是否沉淀到记忆?"或自动写入(根据配置)
    4. 写入对应层级的记忆库,标记来源与置信度
    5. 后续相关会话自动检索并应用
    记忆查看与编辑流程
    1. 用户进入"用户中心 → 我的记忆"
    2. 按层级(用户/项目/会话)浏览记忆条目
    3. 可编辑、删除、导出单条记忆
    4. 可批量导出为 JSON/Markdown 用于审计

    验收标准

    • ✅ 跨会话能延续项目上下文,无需重复说明
    • ✅ 偏好设置后,Agent 响应风格明显改变
    • ✅ 记忆条目可查看、编辑、删除、导出
    • ✅ 每条记忆可追溯到来源(会话 ID 或文档)
    • ✅ 记忆写入/读取全程审计,合规可追溯
    • ✅ 项目隔离,不同项目记忆不互相污染
    • ✅ 时间衰减与主动遗忘机制正常工作
