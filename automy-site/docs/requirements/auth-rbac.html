  • 身份与权限

    • 身份与权限(用户与组织管理)

    面向投资公司全员,提供统一身份认证、组织架构同步、细粒度角色授权与多端会话管理能力,是企业数据主权与最小权限原则的第一道防线。

    🔐
    模块定位
    基于 RBAC + ABAC 双模型,预置 employee / skill-author / admin / auditor 四类核心角色,支持 SSO 单点登录、企业组织架构自动同步、Web 与 Electron 多端会话隔离但共享画像与记忆。

    模块概述

    投资公司组织结构复杂,部门、团队、项目维度交叉,业务员工、技能作者、平台管理员、合规审计员对系统的访问边界差异显著。同时,公司已部署企业 SSO(如 Azure AD / 飞书 / 钉钉 / 自建 OIDC),员工不希望再记忆独立账号密码,IT 部门也要求所有访问行为可统一审计。

    本模块的核心目标是:在不破坏企业现有身份体系的前提下,把 Automy 接入企业 SSO 与组织架构,基于 RBAC + ABAC 实现细粒度授权,并通过 Web 与 Electron 双端会话隔离但画像/记忆共享的机制,保证员工在桌面端与移动办公场景下体验连续,但凭证与权限边界清晰。

    需求条目

    REQ-1.1 SSO 单点登录
    P0

    系统 SHALL 支持 OIDC、SAML 2.0、LDAP 三种主流企业身份协议接入,员工使用企业账号免密登录 Automy;同时 SHALL 提供本地账号密码兜底,以支持开发环境、运维应急与企业 SSO 不可用时的降级登录。

    • 支持协议:OIDC(推荐 Azure AD / 飞书 / 钉钉)、SAML 2.0(对接 OneLogin / ADFS)、LDAP / LDAPS(对接企业 AD)
    • 本地兜底:本地账号 + 密码 + 可选 TOTP 二次验证,仅允许运维角色与开发环境使用
    • 会话管理:登录态保持 8 小时(可配置),支持单点登出与企业 SSO 联动登出
    • JIT 自动开通:首次 SSO 登录时自动创建账号并映射组织信息,无需管理员手动开账号

    场景

    WHEN 投资经理张三使用企业飞书账号首次登录 Automy
    THEN 系统通过 OIDC 协议向飞书 IdP 发起认证,认证成功后自动创建 Automy 账号,同步其部门、邮箱、工号,跳转至工作台首页,无需手动填写注册信息。
    WHEN 企业 SSO 服务因维护不可用
    THEN 系统展示 "SSO 暂不可用,使用本地账号登录" 入口,仅允许具备 admin 角色的运维人员使用本地账号 + TOTP 登录进行应急处理,普通员工登录被拒并提示稍后重试。

    REQ-1.2 组织架构同步
    P0

    系统 SHALL 定期从企业 AD / HR 系统同步组织架构,包括部门、子部门、团队、岗位、汇报关系,作为部门隔离、数据可见范围与审批流转的基础数据;同步频率 SHOULD 不低于每日一次,并支持管理员手动触发即时同步。

    • 同步维度:部门树、岗位、员工信息、上下级汇报关系、跨部门虚拟团队
    • 部门隔离:基于部门 ID 实现数据访问边界,如投研部员工默认无法查看合规部内部知识库
    • 变更检测:员工调岗、离职时自动更新权限与可见范围,离职账号 24 小时内自动停用
    • 冲突处理:本地手动调整与上游同步冲突时,以上游为准并记录变更日志

    场景

    WHEN 投研部员工李四调岗至并购部
    THEN HR 系统次日 02:00 同步后,Automy 自动更新李四的部门归属,撤销其对投研部专属知识库的访问权限,授予并购部知识库访问权限,并通过站内信通知李四与原部门主管。

    REQ-1.3 角色与权限(RBAC + ABAC)
    P0

    系统 SHALL 采用 RBAC(基于角色的访问控制)与 ABAC(基于属性的访问控制)双模型:RBAC 通过预置与自定义角色绑定权限点,ABAC 通过用户/资源/环境属性动态计算访问决策,以满足"投研部经理只能查看本部门项目数据"这类细粒度场景。

    预置角色 SHALL 至少包含以下四类,管理员可基于预置角色派生自定义角色:

    角色典型用户核心权限禁止操作
    employee 普通员工投资经理、研究员个人工作台、订阅专家、安装技能、上传文档查看他人记忆、跨部门访问
    skill-author 技能作者投研骨干、法务专家employee 全部权限 + 创建/编辑/发布技能审批他人技能、查看系统配置
    admin 平台管理员IT 管理员用户管理、角色分配、模型配置、市场审核查看用户记忆内容(仅元数据)
    auditor 合规审计员合规、内审审计日志查看、合规报告导出、配置审查修改业务数据、调用专家
    • 权限点:细粒度权限点(read / write / delete / admin),覆盖工作台、技能、专家、知识库、MCP、记忆等模块
    • 角色继承:支持角色层级,子角色继承父角色权限并叠加自身权限
    • ABAC 属性:用户(部门、职级)、资源(部门归属、密级)、环境(时间、IP、设备)
    • 权限校验:所有 API 与 UI 操作均需经过统一权限网关,失败时返回 403 并记录审计

    场景

    WHEN 投研部经理王五尝试查看并购部某尽调项目的会话记录
    THEN ABAC 引擎检测到用户部门=投研部、资源部门=并购部、密级=项目内部,判定拒绝访问,返回 403 并写入审计日志 "用户越权访问被拒"。

    REQ-1.4 多端会话隔离(共享画像与记忆)
    P1

    系统 SHALL 支持 Web 浏览器与 Electron 桌面客户端两端登录使用,两端会话 Token 相互隔离,任一端登出不影响另一端;但 SHALL 共享同一用户的画像、记忆、订阅、技能安装列表,确保员工在不同设备上获得一致的个性化体验。

    • 会话隔离:Web 与 Electron 各自维护独立 Access Token,刷新 Token 可共享但设备指纹独立
    • 状态同步:用户偏好、订阅、安装的技能、MCP 连接实时云端同步,登录新设备立即可用
    • 并发会话:同一用户最多保持 5 个活跃会话,超出时按最早登录时间踢出
    • 跨端续读:在 Web 创建的会话可在 Electron 打开继续,消息流实时同步
    • 设备管理:用户中心展示所有活跃设备,支持远程登出指定设备

    场景

    WHEN 员工赵六上午在 Web 工作台与 "投研分析师" 专家讨论 ABC 新能源财报,下午出差途中打开 Electron 桌面客户端
    THEN Electron 自动同步上午的会话历史,赵六可在桌面端继续该会话;同时赵六的偏好设置、已订阅专家、已安装技能均同步至桌面端,无需重新配置。
    WHEN 赵六在 Electron 中执行 "退出登录"
    THEN 仅 Electron 的会话 Token 失效,Web 端会话保持登录状态,不影响 Web 端继续使用;赵六下次打开 Electron 需重新 SSO 登录。

    验收标准

    • ✅ 支持 OIDC / SAML / LDAP 至少一种协议接入企业 SSO,本地账号兜底可用
    • ✅ 组织架构每日同步,员工调岗后权限 24 小时内自动调整
    • ✅ 四类预置角色权限边界清晰,ABAC 能基于部门属性拒绝越权访问
    • ✅ Web 与 Electron 会话隔离,共享画像与记忆,跨端续读流畅
    • ✅ 所有登录、权限校验失败、越权访问事件均写入审计日志
